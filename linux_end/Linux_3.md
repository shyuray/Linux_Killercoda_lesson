# Stage 3｜防火牆
<br>

## 這一關的處境
<br>

服務在執行、監聽 `0.0.0.0`、你用真實 IP 也連得到。

前面兩層都排除了。外部還是連不到的話，只剩最後一層。

---
<br>

## 你今天只需要知道這些
<br>

防火牆是**核心裡的一組規則**。每一個進來的封包，核心會拿這組規則逐條比對，決定放行或丟棄。

它跟前兩關的差別在於發生的位置：

```text
封包從網路送到這台機器
      │
      ▼
   防火牆規則 ── 不符合 → 直接丟棄
      │                    （程式完全不知道有人來過）
      ▼ 放行
   核心查對應表：這個 IP + port 有註冊嗎？
      │
      ▼ 有
   交給對應的程式
```

**防火牆在最前面。** 它丟掉的封包，服務連知道都不會知道。所以你查服務的 log，什麼也查不到——對它來說，那個人根本沒來過。

**「服務的 log 乾乾淨淨，但別人就是連不到」——這是防火牆的典型症狀。**

Ubuntu 上管理這組規則的工具叫 **ufw**。

---
<br>

## 一件你必須先知道的事
<br>

**你沒辦法從這台機器自己測試防火牆。**

原因是路由：**你連自己的 IP，封包走的是 loopback，根本不會離開這台機器。**

```text
curl 172.30.1.2        ← 目的地是自己的 IP
      │
      ▼ 核心發現這是本機的位址
      │
   走 loopback，內部直接轉給對應的程式
      │
      ▲
   完全沒經過實體網卡，也沒經過防火牆的 INPUT 規則
```

而 ufw 的預設規則允許 loopback 的所有流量。

**所以你 `ufw deny 80` 之後再 `curl 自己的IP`，它照樣會通。** 那不是防火牆壞了，是那個封包根本沒經過防火牆。

要真的測試防火牆，**必須從另一台機器連過來**。

> 這跟你上一關學的是同一類陷阱。
>
> 上一關：`curl localhost` 通，不代表外部連得到。
> 這一關：`curl 自己的IP` 通，不代表防火牆有放行。
>
> **在本機測出來的結果，永遠不能代表外部的情況。**

<br>

## 角色 / 工具速查表
<br>

| 你想知道 | 打這個 |
| :--- | :--- |
| 防火牆啟用了嗎？放行哪些 port？ | `sudo ufw status` |
| 連編號一起顯示（要刪規則時用） | `sudo ufw status numbered` |
| 放行某個 port | `sudo ufw allow 80` |
| 封鎖某個 port | `sudo ufw deny 80` |
| 刪掉第 N 條規則 | `sudo ufw delete <N>` |
| 啟用 | `sudo ufw enable` |
| 停用 | `sudo ufw disable` |

## 跟著做
<br>

**第一步：先看它現在的狀態。**

`sudo ufw status`{{execute}}

*→ `Status: inactive`。這是 Ubuntu 的預設值。*

**這是一個結論，不是「沒查到」。** 防火牆沒有啟用，代表目前所有封包一律放行，**它不可能是造成連不到的原因**。這個可能性可以從清單上劃掉。

**排除也是進展。**

**第二步：啟用之前，先確認你是怎麼連進這台機器的。**

`sudo ss -tlnp sport = :22`{{execute}}

*→ 那是 sshd。在真實伺服器上，你就是走 port 22 連進來的。*

> **這一步不能跳。**
>
> ufw 啟用之後的預設行為是**拒絕所有外部連入**，包括 port 22。
>
> **而它有一個更陰險的地方：已經建立的連線會繼續存活，它只擋新的連線。**
>
> 所以你 `enable` 之後，畫面一切正常，你以為沒事。**直到你關掉視窗、要重新連進來的那一刻**——那時候你才發現進不去了，而且再也回不來。
>
> 在真實伺服器上，這代表你得請機房的人實體接上去救。

正確的順序：**先放行，再啟用。**

`sudo ufw allow 22`{{execute}}

`sudo ufw allow 80`{{execute}}

`sudo ufw enable`{{execute}}

*→ 它會警告可能中斷現有連線。輸入 `y` 繼續。*

**第三步：看規則。**

`sudo ufw status numbered`{{execute}}

*→ 每一條規則前面有編號。清單上有的才放行，其餘全擋。*

**第四步：封鎖 80，然後觀察一件事。**

`sudo ufw deny 80`{{execute}}

`sudo ufw status numbered`{{execute}}

*→ 規則變了。*

`curl -m 5 $(hostname -I | awk '{print $1}')`{{execute}}

*→ **它還是通。**這正是上面說的：你連自己的 IP，封包走 loopback，沒經過防火牆。*

`systemctl status nginx --no-pager`{{execute}}

*→ 服務也依然 active。*

**這兩個結果都不能證明防火牆有沒有在擋。** 要測，只能從別台機器連。

**第五步：收拾。**

`sudo ufw allow 80`{{execute}}

`sudo ufw disable`{{execute}}

`sudo ufw status`{{execute}}

## 看懂結果
<br>

### `ufw status` 的兩種輸出

| 你看到 | 意思 | 對排查的意義 |
| :--- | :--- | :--- |
| `Status: inactive` | 沒有規則在運作，全部放行 | **它不是原因**，往別處查 |
| `Status: active` + 規則清單 | 只有清單上的 port 允許連入 | 你要連的那個 port 在清單上嗎？ |

### 被防火牆擋住，跟沒有服務在監聽，症狀不一樣

這個差別在真實排查時很有用：

| 情況 | 對方看到的 |
| :--- | :--- |
| **沒有程式註冊那個 port** | 立刻回 `Connection refused` |
| **防火牆 deny（丟棄封包）** | **一直卡住，直到 timeout** |

差在核心怎麼處理：沒人註冊那個 port，核心會回一個「拒絕」的封包告訴對方；防火牆 deny 是**直接把封包丟掉，什麼都不回**，對方只能一直等。

**「立刻被拒絕」和「連很久才 timeout」，是兩種不同的故障。** 前者查監聽，後者查防火牆。

## 常見誤會
<br>

**「我 deny 了但還是連得到，防火牆壞了。」**

你是從機器自己連的。**封包走 loopback，沒經過防火牆。** 這不是防火牆壞了，是你的測試方法測不到它。

**「服務連不到，先關防火牆試試看。」**

關掉之後如果通了，你確實知道原因了。但**在正式環境把防火牆整個關掉，等於把整台機器暴露出去**。正確做法是查規則、只放行需要的那個 port。

**「防火牆擋住的話，服務的 log 會有紀錄吧？」**

不會。封包在到達服務之前就被核心丟掉了。**服務的 log 乾乾淨淨，這本身就是線索。**

**「`ufw enable` 就是啟用而已，我的連線沒斷，很安全。」**

**你的連線沒斷，是因為它已經建立了。** ufw 只擋新連線。你會在下次重連的時候才發現自己被鎖在外面——而那時候已經來不及了。

## 觀念確認（小任務）
<br>

你要在一台正式伺服器上啟用防火牆。你是用 SSH 遠端連進去的。你直接打了 `sudo ufw enable`，沒有先放行 22。畫面一切正常，連線也沒斷。

這代表沒事嗎？

- A. 是，連線沒斷就是沒事
- B. 不是。ufw 只擋新連線，你下次重連就進不去了
- C. 要重開機才知道

**建議答案是 B。**

這是最陰險的一種事故：**它不會當場出事。** 你以為設定成功了，關掉視窗下班。明天早上要連進去的時候，才發現那台機器再也連不上——而且沒有任何辦法遠端救，只能請人實體接上去。

**動手之前先問：這一步會不會把自己鎖在外面。**
