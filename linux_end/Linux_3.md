# Stage 3｜防火牆
<br>

## 這一關的處境
<br>

服務在執行、監聽 `0.0.0.0`、你用真實 IP 也連得到。

前面兩層都排除了。外部還是連不到的話，只剩最後一層。

---
<br>

## 你今天只需要知道這些
<br>

防火牆是**核心裡的一組規則**。每一個進來或出去的封包，核心會拿這組規則逐條比對，決定放行或丟棄。

它跟前兩關的差別在於發生的位置：

```text
封包送到這台機器
      │
      ▼
   防火牆規則  ─── 不符合 → 直接丟棄（程式完全不知道有人來過）
      │
      ▼ 放行
   核心查對應表：這個 IP + port 有註冊嗎？
      │
      ▼ 有
   交給對應的程式
```

**防火牆在最前面。** 它擋掉的封包，服務連知道都不會知道。所以你查服務的 log，什麼也查不到——因為對它來說，那個人根本沒來過。

Ubuntu 上管理防火牆規則的工具叫 **ufw**。

---
<br>

## 角色 / 工具速查表
<br>

| 你想知道 | 打這個 |
| :--- | :--- |
| 防火牆啟用了嗎？放行哪些 port？ | `sudo ufw status` |
| 放行某個 port | `sudo ufw allow 80` |
| 封鎖某個 port | `sudo ufw deny 80` |
| 啟用防火牆 | `sudo ufw enable` |
| 停用防火牆 | `sudo ufw disable` |

---
<br>

## 跟著做
<br>

**第一步：先看它現在的狀態。**

`sudo ufw status`{{execute}}

*→ 你會看到 `Status: inactive`。這是 Ubuntu 的預設值。*

**這是一個結論，不是「沒查到」。** 防火牆沒有啟用，代表目前所有封包一律放行，**它不可能是造成連不到的原因**。這個可能性可以從嫌疑清單上劃掉。

**排除也是進展。**

**第二步：實際看它擋起來是什麼感覺。**

在啟用之前，先看一件事：

`sudo ss -tlnp | grep :22`{{execute}}

*→ 那是 sshd。你連進這台機器就是走 port 22。*

> **這一步不能跳。** ufw 啟用後的預設行為是「拒絕所有外部連入」。
>
> **如果啟用之前沒有放行 22，你就再也連不進這台機器。** 在真實伺服器上，這代表你得請機房的人實體接上去救。
>
> 這不是理論。它是新手最常見的一次性重大事故。

所以順序是：**先放行，再啟用。**

`sudo ufw allow 22`{{execute}}

`sudo ufw allow 80`{{execute}}

`sudo ufw enable`{{execute}}

*→ 它會警告可能中斷現有連線。輸入 `y` 繼續。*

`sudo ufw status`{{execute}}

**第三步：封鎖 80，看會發生什麼。**

`sudo ufw deny 80`{{execute}}

`curl -m 5 $(hostname -I | awk '{print $1}')`{{execute}}

*→ `-m 5` 是最多等 5 秒就放棄，不然它會一直等。*

`systemctl status nginx`{{execute}}

*→ **服務依然是 active。** 它好好的，只是封包到不了它那裡。按 `q` 離開。*

**第四步：收拾。**

`sudo ufw allow 80`{{execute}}

`sudo ufw disable`{{execute}}

`curl $(hostname -I | awk '{print $1}')`{{execute}}

## 看懂結果
<br>

### 被防火牆擋住 vs 沒有服務在聽

| 情況 | curl 的反應 |
| :--- | :--- |
| **沒有程式註冊那個 port** | 立刻回 `Connection refused` |
| **防火牆 deny（丟棄封包）** | **卡住，直到 timeout** |

差別在核心怎麼處理:沒人註冊那個 port，核心會回一個「拒絕」的封包告訴對方；防火牆 deny 是**直接把封包丟掉，什麼都不回**。對方就只能一直等。

**所以「立刻被拒絕」和「一直卡住」是兩種不同的故障。** 前者查監聽，後者查防火牆。

### `ufw status` 的兩種輸出

| 你看到 | 意思 |
| :--- | :--- |
| `Status: inactive` | 沒有規則在運作，全部放行。**它不是原因** |
| `Status: active` + 規則清單 | 只有清單上的 port 允許連入，其餘全擋 |

## 常見誤會
<br>

**「服務連不到，先關防火牆試試看。」**

關掉之後如果通了，你確實知道原因了。但**你在正式環境把防火牆整個關掉，就是把整台機器暴露出去**。正確做法是查規則、只放行需要的那個 port。

**「防火牆擋住的話，log 裡會有紀錄吧？」**

服務的 log 不會有。封包在到達服務之前就被核心丟掉了，**對服務來說那個人根本沒來過**。所以「服務 log 乾乾淨淨，但別人就是連不到」是防火牆的典型症狀。

**「`ufw enable` 就是啟用而已，很安全。」**

它的預設行為是**拒絕所有外部連入**，包括你正在用的 SSH。**先 `allow 22`，再 `enable`。** 這個順序沒有第二種寫法。

---
<br>

## 觀念確認（小任務）
<br>

你要在一台正式伺服器上啟用防火牆。你是用 SSH 遠端連進去的。正確的順序是：

- A. `ufw enable` → `ufw allow 22`
- B. `ufw allow 22` → `ufw enable`
- C. 順序不重要

**建議答案是 B。** A 的話，`enable` 那一瞬間規則就生效了，port 22 被擋，**你下不了第二個指令**——連線斷了，而且再也連不回去。

這跟你之前學的紀律是同一件事：**動手之前先想「這一步會不會把自己鎖在外面」。**
