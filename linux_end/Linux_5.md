# Stage 5｜磁碟怎麼接到檔案樹上
<br>

## 這一關的處境
<br>

每天晚上的備份任務失敗了，log 說 `No space left on device`。

備份寫到 `/mnt/backup`。但你查 `df -h`，那個路徑根本不在清單裡。

前輩說：

> 先搞清楚 `/mnt/backup` 到底是什麼。它是一顆磁碟，還是只是一個空目錄？

---
<br>

## 你今天只需要知道這些
<br>

第一階段你學過：整個系統是一棵從 `/` 長出來的樹。

但**那棵樹不是一顆磁碟**。樹的不同部分，可以放在不同磁碟上。

Windows 的做法是給每顆磁碟一個代號：C:、D:。**Linux 不這樣。Linux 只有一棵樹，磁碟要接進這棵樹的某個目錄底下才能用。**

接的動作叫 **mount**，那個目錄叫 **mount point**。

過程分三層：

**第一層，磁碟本身。** 系統把它當成一個檔案，放在 `/dev/` 底下——`/dev/sda`、`/dev/vda`。這種東西叫 block device。

**第二層，分割區與檔案系統。** 一顆磁碟可以切成幾塊，`/dev/sda1`、`/dev/sda2`。每一塊上面要建立**檔案系統**（ext4 是 Linux 最常見的），決定檔案怎麼在這塊空間裡排放。

**第三層，mount。** 把某個分割區接到樹上的某個目錄。接上去之後，你寫進那個目錄的東西，實際上存在那顆磁碟上。

**沒有 mount 的磁碟，你在檔案樹上完全看不到它。** 它存在，但沒有入口。

## 角色 / 工具速查表
<br>

| 你想知道 | 打這個 |
| :--- | :--- |
| 這台機器有哪些磁碟？接在哪？ | `lsblk` |
| 哪個目錄底下是哪顆磁碟？用多少？ | `df -h` |
| 把磁碟接上去 | `sudo mount <裝置> <目錄>` |
| 拔掉 | `sudo umount <目錄>` |
| 誰在用這個目錄？ | `sudo lsof <目錄>` |
| 開機要自動接哪些？ | `cat /etc/fstab` |
| 照 fstab 全部接一次（驗證用） | `sudo mount -a` |

## 跟著做
<br>

**第一步：從磁碟的角度看。**

`lsblk`{{execute}}

*→ `TYPE` 是 disk 還是 part（分割區），`MOUNTPOINTS` 是它被接到樹的哪裡。空白的代表它沒被接上，你在檔案樹上看不到它。*

**第二步：從檔案樹的角度看。**

`df -h`{{execute}}

*→ `Mounted on` 那一欄是接在樹的哪個目錄。找找看有沒有 `/mnt/backup`。*

> `lsblk` 從磁碟看：有哪些裝置、切成幾塊、接到哪。
> `df` 從樹看：哪個目錄底下是哪顆磁碟、用掉多少。
> **同一件事的兩個方向。**

**第三步：看 `/mnt/backup` 現在是什麼。**

`ls -la /mnt/backup`{{execute}}

*→ 空的。它現在只是一個普通目錄，沒有任何磁碟接在上面。*

**第四步：那顆要接上去的磁碟在哪。**

`ls -lh /opt/backup.img`{{execute}}

*→ 這是一個檔案，但裡面是一個完整的檔案系統。用檔案模擬磁碟來練習，不會動到真的硬碟。*

**第五步：接上去。**

`sudo mount -o loop /opt/backup.img /mnt/backup`{{execute}}

*→ `-o loop` 是「把一個檔案當成磁碟裝置來 mount」。*

**第六步：再看一次兩個角度。**

`df -h /mnt/backup`{{execute}}

`ls -la /mnt/backup`{{execute}}

*→ 現在 `/mnt/backup` 底下有東西了。那些是磁碟上原本就有的檔案。*

**第七步：看開機時會自動接什麼。**

`cat /etc/fstab`{{execute}}

*→ 最後一行是關於 backup 的，但它被 `#` 註解掉了。所以這台機器開機不會自動接它——這就是為什麼你一開始看不到。*

## 看懂結果
<br>

### `/etc/fstab` 的六個欄位

```text
UUID=xxxx-xxxx   /mnt/data   ext4   defaults   0   2
     裝置          接到哪    檔案系統   選項
```

**第一欄用 UUID，不用 `/dev/sdb1`，是有原因的。**

裝置名稱是開機時依偵測順序給的。你多插一顆硬碟，原本的 sdb 可能變成 sdc，fstab 就指到錯的磁碟去了。**UUID 是建立檔案系統時產生的固定編號，不會變。**

查 UUID：

`sudo blkid`{{execute}}

### 為什麼 fstab 危險

**fstab 寫錯會導致開不了機。** 開機時系統照著它去 mount，找不到裝置或選項寫錯，開機流程會停在那裡。

驗證的方式：

```text
sudo mount -a
```

它會照 fstab 全部 mount 一次。**在重開機之前先跑這個，沒報錯才安全。**

這跟你學過的紀律是同一件事：

| 改什麼 | 套用前先驗 |
| :--- | :--- |
| nginx 設定 | `nginx -t` |
| systemd unit | `systemctl daemon-reload` 後看 status |
| fstab | `sudo mount -a` |

**改的是開機必經的路，就一定要有一個「套用前先驗證」的步驟。**

## 常見誤會
<br>

**「`df -h` 沒看到那個路徑，代表那個目錄不存在。」**

不是。`df` 只列出**有磁碟接在上面**的目錄。`/mnt/backup` 存在，只是它現在是個普通目錄，沒有 mount 任何東西。

**「mount 到一個目錄，那個目錄原本的東西會怎樣？」**

**會被蓋住，看不到。** 它們沒有被刪除，只是你現在看到的是新磁碟的內容。umount 之後原本的檔案會再出現。

這個現象很坑：你以為檔案不見了，其實它們還在，只是被遮住。

**「umount 失敗說 target is busy，那就 -f 強制拔。」**

`busy` 代表有程式正在使用那個目錄底下的檔案。**強制拔掉正在寫入的磁碟會弄壞資料。** 先用 `sudo lsof /mnt/backup` 查是誰在用。

## 觀念確認（小任務）
<br>

你在 `/etc/fstab` 加了一行，要讓某顆磁碟開機自動 mount。存檔之後你應該：

- A. 直接重開機測試
- B. 先跑 `sudo mount -a`，確認沒報錯再重開機
- C. 不用測，fstab 很單純

**建議答案是 B。** fstab 寫錯會讓系統開機時卡住。`mount -a` 會照著 fstab 執行一次，**讓你在重開機之前就知道它會不會失敗**。

在雲端主機或遠端伺服器上，開不了機代表你完全碰不到那台機器。這一步花三秒鐘。
