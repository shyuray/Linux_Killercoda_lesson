# Stage 1｜port 被佔用
<br>

## 這一關的處境
<br>

nginx 又停了。你已經有分診習慣，所以先查狀態，再讀 log。

這次 log 裡的訊息，跟你上一階段看過的都不一樣。

---
<br>

## 你今天只需要知道這些
<br>

程式要收網路資料，得先向核心註冊：「送到 port 80 的資料交給我。」

這個註冊動作，程式呼叫的系統函式叫 **`bind()`**。

而核心那張對應表，**一個 port 只能有一筆紀錄**。已經有程式註冊了 port 80，第二個程式再來註冊，核心直接拒絕，回傳錯誤碼 98。

nginx 的 log 就會出現這一句：

```text
nginx: [emerg] bind() to 0.0.0.0:80 failed (98: Address already in use)
```

**這句話跟你上一階段看過的錯誤，性質完全不同。**

設定寫錯的時候，nginx 給你 `default:23` ——檔名、行號都有，你直接去改那一行就好。

這次它只說「port 被佔用」，**不說是被哪個程式佔用**。

原因在 `bind()` 這個函式本身：它失敗時只回傳一個錯誤碼，不夾帶佔用者的資訊。所以 nginx 也不知道是誰佔的，它沒辦法告訴你。

**要查佔用者，得從另一邊問——直接去讀核心那張對應表。**

---
<br>

## 角色 / 工具速查表
<br>

| 你想知道 | 打這個 |
| :--- | :--- |
| 服務為什麼起不來？ | `journalctl -u nginx -n 10` |
| 誰註冊了 port 80？ | `sudo ss -tlnp \| grep :80` |
| 這台機器上所有被佔用的 port | `sudo ss -tlnp` |
| 結束一個程式 | `kill <PID>` |

---
<br>

## 跟著做
<br>

**第一步：確認狀態。**

`systemctl status nginx`{{execute}}

*→ failed。按 `q` 離開。*

**第二步：讀 log，看它說什麼。**

`journalctl -u nginx -n 10`{{execute}}

*→ 找 `Address already in use` 那一句。注意它有沒有告訴你是誰佔的。*

**第三步：去問核心，誰註冊了 port 80。**

`sudo ss -tlnp | grep :80`{{execute}}

*→ 行尾的 `users:(("程式名",pid=數字,fd=數字))` 就是答案。*

**第四步：結束它。把上一步查到的 PID 填進去。**

`kill <PID>`{{copy}}

**第五步：啟動 nginx，照上一階段的順序確認。**

`sudo systemctl start nginx`{{execute}}

`systemctl status nginx`{{execute}}

`sudo ss -tlnp | grep :80`{{execute}}

*→ 最後一行要看到 `users:(("nginx",pid=...))`。port 80 現在註冊給 nginx 了。*

## 看懂結果
<br>

### `ss -tlnp` 的每一欄

```text
LISTEN  0  511  0.0.0.0:80  0.0.0.0:*  users:(("nginx",pid=2416,fd=5))
  │     │   │        │                        │
  │     │   │        │                        └─ 註冊它的程式與 PID
  │     │   │        └─ 監聽的位址與 port
  │     │   └─ backlog：能排隊等待的連線數
  │     └─ 接收佇列
  └─ 狀態
```

`backlog` 這個數字順帶說明一件事：nginx 是 511，python 的 `http.server` 只有 5。**那是為什麼 python 的簡易伺服器不能拿來當正式服務**——同時來六個人就開始排不進去了。

### 錯誤訊息告訴你什麼、沒告訴你什麼

| 這一階段的錯誤 | 它給你 | 它不給你 |
| :--- | :--- | :--- |
| `invalid parameter in default:23` | 檔名、行號、什麼錯 | —— |
| `Address already in use` | port 被佔用 | **是哪個程式佔的** |

**有些錯誤訊息會指出原因，有些只會指出症狀。** 後者你得自己多走一步。

## 常見誤會
<br>

**「服務起不來，一定是設定或權限的問題。」**

這就是為什麼要讀 log 再動手。設定對、權限對，port 被別人佔走，它照樣起不來。

**「`ss -tlnp` 跑了，但沒看到程式名稱，所以查不到。」**

`-p` 這個參數只顯示**你自己執行的程序**。服務多半是 root 執行的，所以你要加 sudo。**它不會報錯，只是那一欄空白。** 你如果不知道它該長什麼樣，就會以為「沒有這個資訊」。

**「用 `ps aux | grep` 也查得到吧？」**

查得到那個程式在跑，但**查不到它佔用哪個 port**。`ps` 看的是程序，`ss` 看的是網路。要回答「誰佔了 80」，只有 `ss`。

## 觀念確認（小任務）
<br>

服務啟動失敗，log 說 `Address already in use`。你的下一步是：

- A. 檢查設定檔有沒有寫錯
- B. 用 `sudo ss -tlnp` 查是哪個程式佔用了那個 port
- C. 重開機

**建議答案是 B。** 這個錯誤已經明確告訴你原因是 port 被佔用，設定檔沒有問題，查它是浪費時間。而錯誤訊息不會告訴你是誰佔的，你得自己查。

（重開機確實能解決——如果那個佔用的程式不會跟著開機啟動的話。但你不會知道原因，下次還會發生。）
