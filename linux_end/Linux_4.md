# Stage 4｜這台機器連不出去
<br>

## 這一關的處境
<br>

前面三關都是「別人連不到我」。這一關方向相反：**這台機器連不出去。**

症狀可能長得一模一樣（都是「連不到」），但原因和工具完全不同。

前輩說：

> 「網路壞了」這句話，一半的情況其實是 DNS。而分辨只要一個動作。

---
<br>

## 你今天只需要知道這些
<br>

網路上傳輸資料時，寫在封包上的是 IP 數字。**核心和網路設備只認數字，不認名稱。**

但你打的是 `curl google.com`。中間必須有一步：**把名稱換成 IP**。

這一步的做法是向一台 **DNS server** 發出查詢——「google.com 的 IP 是多少？」拿到回答，才帶著那個 IP 去連線。

所以「連不到 google.com」有兩種完全不同的失敗：

| 失敗在哪 | 錯誤訊息 | 意思 |
| :--- | :--- | :--- |
| **名稱換不到 IP** | `Could not resolve host` | 連線根本還沒開始 |
| **連線本身失敗** | `Connection refused` / timeout | IP 拿到了，但連不上 |

**`resolve` 這個字就是分界線。** 看到它，你查 DNS；沒看到它，你查網路。

而分辨的動作只有一個：**直接用 IP 連連看，跳過名稱查詢那一步。**

- IP 連得通、名稱連不通 → **DNS 的問題，網路本身沒事。**
- IP 也連不通 → **網路的問題，DNS 不是原因。**

---
<br>

## 角色 / 工具速查表
<br>

| 你想知道 | 打這個 |
| :--- | :--- |
| 名稱換得到 IP 嗎？ | `nslookup google.com` |
| 只要 IP，不要其他 | `dig google.com +short` |
| 實際是哪台 DNS 在回答？ | `resolvectl status` |
| 這台機器有 IP 嗎？ | `ip a` |
| 封包要往哪送？ | `ip r` |
| 那台代轉的機器活著嗎？ | `ping -c 3 <gateway>` |

---
<br>

## 跟著做
<br>

**第一步：先用名稱連。**

`curl -I -m 5 google.com`{{execute}}

*→ `-I` 只要回應標頭，不要整頁 HTML。`-m 5` 最多等 5 秒。*

**第二步：查出它的 IP。**

`dig google.com +short`{{execute}}

*→ 拿到一個 IP。這證明名稱換 IP 這一步是通的。*

**第三步：直接用 IP 連，跳過名稱查詢。**

`curl -I -m 5 $(dig google.com +short | head -1)`{{execute}}

*→ 兩個都通，代表 DNS 和網路都沒問題。*

> **這一步的重點是它證明了什麼。** 用名稱連跟用 IP 連，中間差的就是「名稱換 IP」那一步。兩個都通，那一步就沒問題；名稱不通、IP 通，問題就鎖定在那一步。

**第四步：看是誰在回答你的 DNS 查詢。**

`nslookup google.com`{{execute}}

*→ 注意 `Server:` 那一行。**記下那個位址，下一步要用。***

**第五步：看這台機器實際設定的 DNS。**

`resolvectl status`{{execute}}

*→ 找兩樣東西：`resolv.conf mode:` 和 `DNS Servers:`。*

**這兩步的答案可能一致，也可能不一致。** 為什麼不一致，下面解釋。

**第六步：封包要往哪送。**

`ip r`{{execute}}

**第七步：測 gateway。**

`ip r | grep default`{{execute}}

*→ `default via` 後面那個 IP 就是 gateway。把它填進下面。*

`ping -c 3 <gateway的IP>`{{copy}}

## 看懂結果
<br>

### `nslookup` 說的 Server，可能不是真正的 DNS

現在的 Ubuntu 有一個叫 `systemd-resolved` 的程式，它在本機開了一個 DNS 窗口。你可以看到它：

`sudo ss -tlnp sport = :53`{{execute}}

*→ 它註冊在 `127.0.0.53` 這個位址上。*

它自己不知道 google.com 的 IP。**它只是一個中間人**：收到查詢之後，轉發給真正的外部 DNS server，拿到答案再回給你。中間這一層的用途是**快取**——同一個名稱問第二次，它直接回答，不用再往外問。

但你的查詢會不會經過它，**取決於 `/etc/resolv.conf` 指向哪裡**。而 `resolvectl status` 裡的 `resolv.conf mode` 就是在講這件事：

| resolv.conf mode | 意思 | `nslookup` 的 Server 會是 |
| :--- | :--- | :--- |
| **stub** | 查詢先送到本機的 systemd-resolved，再由它轉發 | `127.0.0.53` |
| **uplink** | 查詢直接送到外部 DNS，不經過中間人 | 外部 DNS 的 IP（例如 `8.8.8.8`） |

**兩種都正常，只是設定不同。** 你剛才 `nslookup` 看到的 Server，對照 `resolvectl status` 的 mode，應該就對得起來。

排查時的意義：

- 看到 `127.0.0.53` → 中間有一層本機的轉發程式。**它壞了，或它的上游壞了，症狀一樣。** 你要用 `resolvectl status` 才看得到上游是誰。
- 看到外部 IP → 沒有中間層，`nslookup` 顯示的就是真正在回答的那一台。

> **一個實務落差要注意。** 舊教材會叫你改 `/etc/resolv.conf` 來換 DNS。
>
> 但那個檔案現在是自動產生的，**你改了會被覆蓋掉**。要真的改 DNS，得改 netplan 或 resolved 的設定。

### `ip r` 每一行在說什麼

```text
default via 10.0.2.2 dev enp0s3        ← 不符合其他規則的，全交給 10.0.2.2
10.0.2.0/24 dev enp0s3 scope link      ← 這個範圍內的，直接送，不用經過誰
```

**有 `via` 就是「交給那台機器代轉」，沒有 `via` 就是「直接送達」。**

第二行的 `10.0.2.0/24` 是你自己所在的網段。目的地落在 `10.0.2.0` 到 `10.0.2.255` 之間，就直接送，因為對方跟你在同一個網路裡。

第一行的 `default` 是「以上規則都不符合時」用的。`via 10.0.2.2` 代表把封包交給那台機器，由它負責繼續往外送。**那台機器叫 gateway。**

**核心比對路由表時，取最精確的那一條。** 有專屬規則就用專屬的，沒有才落到 default。

**沒有 `default` 那一行**的結果是：同網段的機器連得到，網路外的一律送不出去——核心不知道該把封包交給誰。

### ping 的陷阱

很多機器和防火牆設定成不回應 ping。**ping 不通不代表機器死了**，只代表它不回應 ping。

**ping 適合用來確認「通」，不適合用來斷定「不通」。**

## 常見誤會
<br>

**「網路壞了。」**

這句話什麼資訊都沒有。先問：是名稱換不到 IP，還是 IP 也連不上？一個 `curl <IP>` 就分得出來，**而且一半的情況會在這裡結案。**

**「ping 不通就是機器掛了。」**

不一定。很多機器根本不回應 ping。要確認某個服務通不通，測那個 port，不要測 ping。

**「DNS 是 127.0.0.53，那 DNS server 就在我機器上，改本機設定就好。」**

`127.0.0.53` 只是本機的轉發程式，真正的上游在 `resolvectl status` 裡。**DNS 出問題時，要分清楚是本機的轉發程式壞了，還是上游那台不回應。**

## 觀念確認（小任務）
<br>

同事說「網路壞了，網頁都打不開」。你只能請他跑一個指令。你會請他跑哪一個？

- A. `ping google.com`
- B. `curl -I 1.1.1.1`
- C. `ip a`

**建議答案是 B。**

`ping google.com` 同時用到 DNS 和網路，失敗了你分不出是哪一個壞。而且很多機器不回應 ping，你會得到誤導的結果。

`curl -I 1.1.1.1` 用 IP 直連一個確定有跑網頁的位址，**它一刀切開 DNS 和網路**：通了就是 DNS 的問題，不通就是網路的問題。**一個指令砍掉一半的可能性。**

（`ip a` 有用，但那是確定網路有問題之後才查的下一層。）
