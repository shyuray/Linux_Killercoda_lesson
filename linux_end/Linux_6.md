# Stage 6｜還有空間，卻寫不進去
<br>

## 這一關的處境
<br>

`/mnt/backup` 接上去了。`df -h` 說它只用了 1%，空間多得很。

但備份任務還是失敗，錯誤訊息一模一樣：

```text
No space left on device
```

系統說沒空間，`df` 說有空間。**有一邊在說謊，或者你查錯了東西。**

---
<br>

## 你今天只需要知道這些
<br>

檔案系統存兩種東西：

**檔案的內容**——你寫進去的資料。

**檔案的資料**——名字、擁有者、權限、時間、以及內容實際放在磁碟的哪幾個位置。這種紀錄叫 **inode**。

每建立一個檔案，就佔用一個 inode。**不管那個檔案是 1 byte 還是 10GB，都只佔一個。**

關鍵在這裡：**inode 的總數，在建立檔案系統的時候就固定了，之後不能增加。**

所以「滿了」有兩種：

| 哪一種滿了 | 意思 | 查法 |
| :--- | :--- | :--- |
| **空間滿了** | 沒有地方放內容 | `df -h` |
| **inode 滿了** | 沒有地方登記新檔案 | `df -i` |

**而它們的錯誤訊息一模一樣，都是 `No space left on device`。**

這就是為什麼會出現「`df -h` 明明還有空間，系統卻說沒空間」——**你查的是空間那個維度，但滿掉的是另一個。**

---
<br>

## 角色 / 工具速查表
<br>

| 你想知道 | 打這個 |
| :--- | :--- |
| 空間還剩多少？ | `df -h` |
| inode 還剩多少？ | `df -i` |
| 哪個目錄佔最多**空間**？ | `sudo du -sh /path/* \| sort -rh` |
| 哪個目錄有最多**檔案**？ | `find` 加管線（下面示範） |

---
<br>

## 跟著做
<br>

**第一步：先確認空間。**

`df -h /mnt/backup`{{execute}}

*→ 用不到 1%。空間根本沒滿。*

**第二步：試著寫一個檔案進去。**

`touch /mnt/backup/test.txt`{{execute}}

*→ `No space left on device`。**空間有 3MB，但一個空檔案都建不出來。***

**第三步：查另一個維度。**

`df -i /mnt/backup`{{execute}}

*→ `IUse%` 是 100%。`IFree` 是 0。*

**這兩個指令的輸出擺在一起看：**

```text
df -h：  Size 4.0M   Used 24K    Avail 3.4M    Use%   1%     ← 空間爆多
df -i：  Inodes 32   IUsed 32    IFree 0       IUse% 100%    ← 登記簿滿了
```

**空間夠，但沒有地方登記這個新檔案。**

**第四步：看是什麼東西吃掉的。**

`ls /mnt/backup`{{execute}}

*→ 一堆 `session_*.tmp`。每個都是 0 byte，加起來一點空間都不佔。**但每一個都吃掉一個 inode。***

**第五步：清掉它們。**

`sudo rm /mnt/backup/session_*.tmp`{{execute}}

`df -i /mnt/backup`{{execute}}

`touch /mnt/backup/test.txt`{{execute}}

*→ inode 放出來了，現在寫得進去。*

---
<br>

## 找出誰吃掉 inode
<br>

空間滿了，你用 `du -sh` 找**大檔案**。

inode 滿了，`du` 沒用——**問題不在大檔案，在檔案數量。** 幾百萬個 1KB 的檔案，`du` 算出來微不足道，但每一個都吃掉一個 inode。

要找「哪個目錄底下檔案數最多」：

`sudo find /var -xdev -type f | sed 's|/[^/]*$||' | sort | uniq -c | sort -rn | head`{{execute}}

拆開看：

| 這一段 | 做什麼 |
| :--- | :--- |
| `find /var -xdev -type f` | 找出所有檔案。`-xdev` 是不要跨到別的 mount point |
| `sed 's\|/[^/]*$\|\|'` | 把路徑最後的檔名切掉，只留目錄 |
| `sort \| uniq -c \| sort -rn` | 排序、計數、排名 |

**後面那三段，跟你上一階段算攻擊來源 IP 排名，是同一條管線。**

當時你學的不是一條指令，是一個模式：**撈出來 → 排序 → 計數 → 排名**。換掉前面撈的東西，它就能回答不同的問題。

## 看懂結果
<br>

### 兩種「滿了」的對照

| | 空間滿 | inode 滿 |
| :--- | :--- | :--- |
| **錯誤訊息** | `No space left on device` | `No space left on device`（**一樣**） |
| **查法** | `df -h` | `df -i` |
| **佔用者的樣子** | 少數幾個巨大檔案 | 幾十萬個小檔案 |
| **怎麼找** | `du -sh` 排序 | `find` + 管線數檔案數 |
| **常見來源** | log 沒清、上傳檔案 | session 檔、快取、郵件佇列 |

### 實務上什麼東西會吃光 inode

session 檔、快取檔、郵件佇列、沒清的暫存檔。它們每個只有幾 KB，**加起來的空間微不足道，但每一個都佔一個 inode**。

一台跑了三年的機器，`/tmp` 或 `/var/spool` 底下累積幾十萬個小檔案，是很常見的事。

<br>

## 常見誤會
<br>

**「`df -h` 說有空間，那就不是磁碟的問題。」**

這是這一關的全部。**「滿了」有兩種，你只查了一種。** 看到 `No space left on device`，`df -h` 和 `df -i` 兩個都要看。

**「inode 用完了，那就把檔案系統擴大。」**

擴大空間不會增加 inode。**inode 的數量在建立檔案系統時就決定了**，事後改不了。要增加只能重建檔案系統——那代表資料要先搬走。

所以正確的做法是清掉不需要的小檔案，而不是加硬碟。

**「rm 掉那些檔案就好了。」**

方向對，但先搞清楚那些檔案是什麼、還有沒有人在用。`session_*.tmp` 這種名字看起來可以刪，但如果它是某個服務正在用的，刪了會出事。**這跟你之前學的「先鎖定、蒐證，再移除」是同一個原則。**

## 觀念確認（小任務）
<br>

同事說「磁碟明明還有 50GB，但服務一直說 No space left on device」。你會請他跑哪個指令？

- A. `du -sh /*`，找出最大的目錄
- B. `df -i`，查 inode
- C. 重開機

**建議答案是 B。**

他已經告訴你「還有 50GB」了，代表他查過 `df -h`，空間確實沒滿。**那就只剩另一個維度。**

A 是查空間的工具，而空間已經被排除了，查它是浪費時間。

---
<br>

## 收拾
<br>

`sudo umount /mnt/backup`{{execute}}

`df -h | grep backup`{{execute}}

*→ 沒有輸出，代表已經拔掉了。*
