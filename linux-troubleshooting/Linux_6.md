# Stage 6｜追查：是誰在嘗試登入？
<br>

## 這一關的處境
<br>

服務恢復了。你在清理紀錄的時候，注意到 `/var/log/auth.log` 裡有一批登入失敗的紀錄——不同的來源位址、不同的帳號、密碼一直錯。

前輩湊過來看了一眼：

> 幾千行，你打算一行一行看？
>
> 這裡有三個問題要回答：是誰在試？試了幾次？他們鎖定哪個帳號？
>
> 你不能用眼睛看，你要用算的。而且不要憑印象猜——你猜錯的地方，就是攻擊者藏身的地方。

---
<br>

## 你今天只需要知道這些
<br>

前面五關，你在**修東西**。這一關開始，你在**查東西**。同樣是讀執行紀錄，目的完全不同。

修東西的時候，你要找的是**一句話**：某個 error，它直接說明服務為什麼停。

查東西的時候，你要找的是**規律**：誰出現最多次、什麼事情最頻繁。

規律不會浮在表面。你需要的答案是「某個位址失敗了 47 次」，但紀錄只會給你 47 行分散在各處的資料。所以你要學會一種做法：**把一堆文字，變成一張排名表。**

### 管線：把小工具接成一條生產線

Linux 的設計是每個工具只做好一件事，然後用管線符號把它們接起來，前一個的輸出變成後一個的輸入。

```text
   一堆執行紀錄
      │
      ▼  grep      撈出我關心的那些行，濾掉雜訊
      │
      ▼  awk       只取出我要的那一欄（例如來源位址）
      │
      ▼  sort      把相同的排在一起
      │            ┌────────────────────────────────┐
      │            │ uniq 只能合併「相鄰」的重複行     │
      │            │ 所以必須先 sort，否則數不出來    │
      │            └────────────────────────────────┘
      ▼  uniq -c   合併相鄰的重複，並數出次數
      │
      ▼  sort -rn  依次數排名，最多的排最上面
      │
   一張排名表
```

**`sort` 不是可有可無的一步，它是 `uniq -c` 能運作的前提。**

`awk` 本身是一個完整的程式語言，今天只用它最基本的一招：把每一行按空白切開，取出第 N 欄。

## 角色 / 工具速查表
<br>

| 你想做 | 打這個 |
| :--- | :--- |
| 撈出含某個關鍵字的行 | `grep "Failed password" 檔案` |
| 只取第 11 欄 | `awk '{print $11}'` |
| 一次取兩欄 | `awk '{print $9, $11}'` |
| 把相同的排在一起 | `sort` |
| 數出重複次數 | `uniq -c` |
| 依數字大小排名 | `sort -rn` |
| 只看排名前 5 | 在管線**最後**接上 `head -n 5` |
| 看命中行的前後文 | `grep -B 3 -A 1 "關鍵字" 檔案` |
| 濾掉權限錯誤訊息 | 在指令後加上 `2>/dev/null` |

> **怎麼知道位址在第幾欄？自己數。**
>
> 把一行按空白切開：
>
> `Jul`(1) `10`(2) `01:22:11`(3) `srv`(4) `sshd[1001]:`(5) `Failed`(6) `password`(7) `for`(8) `root`(9) `from`(10) `203.0.113.45`(11)
>
> 帳號在第 9 欄，來源位址在第 11 欄。換一種紀錄格式，欄位就不一樣——**數欄位是這份工作的日常，不需要背。**

## 跟著做
<br>

一層一層加上去，每加一層，看它多做了什麼。

**第 1 層：撈出所有登入失敗的紀錄。**

`sudo grep "Failed password" /var/log/auth.log`{{execute}}

*→ 成功登入的行被濾掉了，只剩失敗的。從雜訊裡挑出訊號。*

**第 2 層：只取出來源位址。**

`sudo grep "Failed password" /var/log/auth.log | awk '{print $11}'`{{execute}}

*→ 現在只剩一串位址，其他文字都不見了。*

**第 3 層：排序，為了下一步的計數做準備。**

`sudo grep "Failed password" /var/log/auth.log | awk '{print $11}' | sort`{{execute}}

*→ 相同的位址現在相鄰了。*

**第 4 層：計數。**

`sudo grep "Failed password" /var/log/auth.log | awk '{print $11}' | sort | uniq -c`{{execute}}

*→ 每個位址前面多了一個數字，代表它出現幾次。*

**第 5 層：排名。**

`sudo grep "Failed password" /var/log/auth.log | awk '{print $11}' | sort | uniq -c | sort -rn`{{execute}}

*→ 一張來源排名表。最上面那個，就是嘗試最多次的。*

**換一個問題：他們鎖定哪個帳號？**

管線結構完全不動，只把 `$11` 換成 `$9` ：

`sudo grep "Failed password" /var/log/auth.log | awk '{print $9}' | sort | uniq -c | sort -rn`{{execute}}

**最後：有沒有人真的登入成功了？**

`sudo grep -B 3 -A 1 "Accepted password" /var/log/auth.log`{{execute}}

*→ `-B 3` 是命中行的前 3 行，`-A 1` 是後 1 行。重點是看那次成功登入的前後，發生了什麼。*

## 看懂結果
<br>

### 1. 排名表怎麼讀

```text
     47 203.0.113.45   ← 嘗試 47 次，主要來源
      8 198.51.100.7   ← 次要
      1 203.0.113.99   ← 可能只是打錯密碼的同事
```

次數的差距本身就是情報。47 次不是手滑，是自動化工具在跑字典。1 次很可能只是有人記錯密碼。

### 2. 為什麼要看前後文

單獨看一行 `Accepted password` ，完全正常——每天都有人成功登入。

但如果它夾在一連串失敗嘗試中間，意義就變了：

```text
Failed password for root ...       ← 失敗
Failed password for admin ...      ← 失敗
Failed password for root ...       ← 失敗
Accepted password for deploy ...   ← 成功
                                      這次成功很可疑
```

**一個事件可不可疑，取決於它周圍發生了什麼。** 這就是 `-A` 和 `-B` 存在的理由。

> **寫進報告的時候：**
>
> 不能寫「我覺得怪怪的」，要寫：
>
> *「在該次成功登入之前的短時間內，同一來源出現多次密碼失敗紀錄，符合暴力破解成功的模式，應進一步查證來源位址與該帳號的後續行為。」*
>
> **直覺是起點，證據才是交付物。**

## 常見誤會
<br>

**「攻擊者一定是在試 root，那是最高權限。」**

也許。但攻擊者知道你在盯 root，所以他們可能改試 admin、deploy、test。**你的假設，就是他們的藏身處。** 用管線算出來，不要猜。

**「`head -n 5` 放在管線中間也一樣。」**

差很多。

放在**中間**，是把資料砍掉——只讓前 5 行流下去，後面的統計建立在殘缺的樣本上，結論會是錯的。

放在**最後**，是只砍顯示——統計在完整資料上做完，只是最後只列出前 5 名。

**放中間是砍資料，放最後是砍顯示。**

**「輸出跑出來了，那就是對的。」**

輸出看起來合理，不代表指令邏輯正確。少了 `sort` 、`head` 放錯位置、欄位數錯——這些都會產生**看起來正常、但完全錯誤**的數字。

**一份被截斷過的統計，比沒有統計更危險，因為它看起來有數據支撐。**

## 觀念確認（小任務）
<br>

你想統計哪個來源位址登入失敗最多次，打了這條指令，但少了 `sort` ：

```text
grep "Failed password" auth.log | awk '{print $11}' | uniq -c | sort -rn
```

結果出現一堆 1 和零星的 2，沒有任何一個位址次數很高。最可能的原因是：

- A. 這台機器沒有被攻擊
- B. 少了 sort，相同的位址沒有排在一起，uniq -c 數不到它們
- C. awk 的欄位取錯了

**建議答案是 B。** `uniq -c` 只合併**相鄰**的重複行。沒有先 sort 把相同的位址排在一起，散落各處的同一個位址就不相鄰，自然數不出來。
