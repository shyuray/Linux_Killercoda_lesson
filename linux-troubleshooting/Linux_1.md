# Stage 1｜它現在有沒有在執行？
<br>

## 這一關的處境
<br>

你連進伺服器。機器沒關機，指令打下去都有回應，表面上看不出問題。但同事說網頁一直打不開。

你想先開設定檔看看哪裡寫錯。前輩攔住你：

> 別急著改東西。
>
> 第一個問題不是「怎麼修」，是「它到底還在不在跑」。
>
> 這兩件事的處理方式完全不同。

**機器還開著，不代表它上面的每個服務都還在執行。** 這是兩件事。在確認服務的實際狀態之前，任何修改都只是猜測。

---
<br>

## 你今天只需要知道這些
<br>

任何時候去看，一個服務只會是這三種狀態的其中一種：

| 狀態 | 實際發生了什麼 | 這代表 |
| :--- | :--- | :--- |
| **執行中** | 正常運作，隨時處理請求 | 問題不在這個服務，往別處查 |
| **啟動失敗** | 它試著啟動，中途出錯停住 | 有東西讓它跑不起來 |
| **被停止** | 有人手動把它關掉 | 這不是故障，是有人主動關的 |

分清楚是哪一種，決定了你接下來做什麼。

誤把「被停止」當成「啟動失敗」，你會去翻一份根本沒問題的設定檔。誤把「啟動失敗」當成「執行中」，你會去查一條根本沒斷的網路。

---
<br>

## 看圖理解
<br>

要判斷它是哪一種狀態，你得先知道該問誰。

這台機器上所有服務的啟動與停止，統一由 systemd 管理。`systemctl` 是你對 systemd 下令的指令。

```text
                        ┌──▶  nginx   （提供網頁）
                        │
 你 ──systemctl──▶  systemd  ──┼──▶  ssh     （遠端連線）
                        │
                        └──▶  cron    （排程工作）
```

從左看起：你手上只有 systemctl 這一個管道；中間的 systemd 掌握所有服務的狀態；右邊才是實際在做事的服務。

**你不直接碰服務，你透過 systemd 去問、去下令。**

---
<br>

## 角色 / 工具速查表
<br>

| 你想知道 | 打這個 |
| :--- | :--- |
| 這台機器上有哪些服務啟動失敗？ | `systemctl --failed` |
| 某個服務現在是什麼狀態？ | `systemctl status nginx` |
| 系統裡到底有沒有這支程式在跑？ | `ps aux \| grep nginx` |
| 網站真的會回應嗎？ | `curl localhost` |

> **提示：** `systemctl --failed` 是維運人員每天的第一個動作。
>
> 它一次掃過整台機器，直接列出所有出狀況的服務，不必逐一去猜、逐一去查。

---
<br>

## 跟著做
<br>

查東西有先後：先問範圍最大的，再逐步縮小到單一服務。這個由大到小的順序，就是分診本身。

**第一步：先掃全機，看有沒有服務出狀況。**

`systemctl --failed`{{execute}}

*→ 你應該看到 nginx 出現在清單裡，狀態是 failed。災情確認。*

**第二步：範圍縮小了，單獨查這個服務。**

`systemctl status nginx`{{execute}}

*→ 輸出很長。先只看第三行的 `Active:` ，其餘略過。看完按 `q` 離開。*

**第三步：換個角度驗證。從外部發一個請求，看它會不會回應。**

`curl localhost`{{execute}}

*→ 應該出現 `Connection refused` 。*

**第四步：直接翻程式清單，確認它在不在。**

`ps aux | grep nginx`{{execute}}

*→ 你會看到一行結果。先別急著下結論——仔細看那一行，它是 nginx 服務，還是你自己剛下的那道搜尋指令？*

---
<br>

## 看懂結果
<br>

這四個指令都會回你一大片文字。你不需要每個字都讀懂，只要認得其中的關鍵字。

### 1. `systemctl status` 的 `Active:` 那一行

| 你看到 | 這句話的意思 | 下一步 |
| :--- | :--- | :--- |
| **active (running)** | 服務正在執行 | 問題不在這裡，往別處查 |
| **failed** | 它試著啟動，出錯停在半路 | 往下走，去看它的執行紀錄 |
| **inactive (dead)** | 它被正常關掉了，沒有出錯 | 這不是故障，直接啟動即可 |

### 2. `curl localhost` 的結果

| 你看到 | 這句話的意思 |
| :--- | :--- |
| **一大段 HTML** | 網站有回應，服務正常 |
| **Connection refused** | 連線被拒絕：沒有任何程式在那個位置接收請求 |

兩邊的答案是兜得起來的。`status` 說 failed，`curl` 說連線被拒絕——兩個指令從不同角度指向同一件事。這不是誤判，服務確實沒在執行。

---
<br>

## 常見誤會
<br>

**「用 ps 找得到 nginx，所以服務還在跑。」**

你下的那道搜尋指令，本身的文字裡就含有 nginx 這幾個字，所以會把自己也搜出來。不要把自己的搜尋動作當成服務還在執行的證據。真正的服務程式，那一行會出現 `nginx: master process` 這樣的字樣。

**「網頁打不開，一定是網路問題。」**

網路正常但服務停止，跟服務正常但網路不通，從瀏覽器上看起來一模一樣，處理方式卻完全不同。**先確認服務，再懷疑網路。**

**「服務停了，一定是這個服務本身有問題。」**

也可能是磁碟滿了、檔案權限不對、或它要用的位置被別的程式占走。服務只是最後倒下的那一個，未必是出問題的那一個。這就是為什麼要先分診。

## 觀念確認（小任務）
<br>

你執行 `systemctl restart nginx` ，終端機沒有跳出任何錯誤訊息。這代表服務已經正常運作了嗎？

- A. 是，沒報錯就是成功
- B. 不一定，要再用 `status` 確認
- C. 要重開機才知道

**建議答案是 B。**

`systemctl` 沒報錯，只代表你的指令被接受了，不代表服務真的起來。服務可能在背景嘗試啟動、失敗、重試，反覆幾次後放棄，而終端機這邊完全看不出來。

**指令沒有報錯，不等於服務正在執行。**
