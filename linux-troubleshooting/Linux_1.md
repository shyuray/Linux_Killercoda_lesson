# Stage 1｜它還活著嗎？
<br>

## 這一關的處境
<br>
你 SSH 進伺服器。畫面正常，機器沒關機，你打指令它都有回應。

但同事說網頁一直轉圈圈。

前輩在旁邊說：

> 別急著改東西。
>
> 第一個問題不是「怎麼修」，是 **「它到底還在不在跑」** 。
>
> 這兩件事的處理方式完全不同。

---
<br>

## 你今天只需要知道這些

一個服務只會處在三種狀態之一，你要分得出來：

| 狀態 | 白話 | 意義 |
| :--- | :--- | :--- |
| **活得好好的** | 它正在工作 | 那問題不在服務，在別處 |
| **出事死了** | 它自己倒下的 | 有東西讓它啟動失敗 |
| **被正常關掉了** | 有人手動停了它 | 不是崩潰，是有人關的 |

*（它有幾種狀態機、怎麼轉換，今天不用管。）*

---
<br>

## 看圖理解

看圖順序：先看**左邊**，你手上只有一支對講機；**中間**是總管家，它掌握所有服務的生死；**右邊**才是實際幹活的服務。

```text
                    ┌──▶  nginx   （公司內部系統）
                    │
 你 ──systemctl──▶ systemd ──┼──▶  ssh     （遠端連線）
    （對講機）      （總管家）  │
                    └──▶  cron    （排程工作）
```

**你不直接碰服務，你跟總管家下指令。**

---
<br>

## 角色 / 工具速查表

| 你想知道 | 打這個 |
| :--- | :--- |
| 這台機器上有哪些服務掛了？ | `systemctl --failed` |
| 某個服務現在活著嗎？ | `systemctl status nginx` |
| 到底有沒有這個程序在跑？ | `ps aux \| grep nginx` |
| 網站真的有回應嗎？ | `curl localhost` |

> **提示：** `systemctl --failed` 是維運巡檢的第一動作。
>
> 一眼看完全機有哪些服務死了，不用一個一個查。

---
<br>

## 跟著做

請依序點擊以下指令，觀察右邊終端機給你的反饋。

**第一步：全機掃描，看看誰掛了。**

`systemctl --failed`{{execute}}

*→ 你應該看到 nginx 出現在清單裡，標記為 failed。這就是災情確認。*

**第二步：單獨盤問這個服務。**

`systemctl status nginx`{{execute}}

*→ 畫面很長，只看第三行 `Active:` ，其他先略過。按 `q` 離開。*

**第三步：從外面敲門，看有沒有人應答。**

`curl localhost`{{execute}}

*→ 應該出現 `Connection refused` 。沒人在門口接客。*

**第四步：看看程序清單裡到底有沒有它。**

`ps aux | grep nginx`{{execute}}

*→ 你會看到一行結果。仔細看那行 —— 它是 nginx 服務嗎？還是你自己剛剛下的那個搜尋指令？*

## 看懂結果

### 1. `systemctl status` 的 `Active:` 那一行

| 你看到 | 白話翻譯 | 下一步 |
| :--- | :--- | :--- |
| **● active (running)** | 活著，正在工作 | 問題不在服務，查別處 |
| **× failed** | 出事死了 | 往下走，去讀它的遺言 |
| **inactive (dead)** | 有人正常把它關掉了 | 不是崩潰，直接啟動它 |

### 2. `curl localhost` 的結果

| 你看到 | 白話翻譯 |
| :--- | :--- |
| **一堆 HTML** | 網站有回應，服務正常 |
| **Connection refused** | 沒人在那個門口接客 —— 服務沒跑 |

## 常見誤會

**「用 ps 有找到 nginx，所以服務還活著。」**

搜尋指令本身的文字裡就含有 nginx 這個字，所以它會把自己也搜出來。**別把你的搜尋動作，當成服務還活著的證據。** 要找的是有 `nginx: master process` 這種字樣的那一行。

**「網頁打不開，一定是網路問題。」**

網路正常、服務死了，結果一樣打不開。**先確認服務，再懷疑網路。**

**「服務掛了一定是服務本身壞掉。」**

也可能是磁碟滿了、權限錯了、或門口被別人佔走了。**這就是為什麼要分診。**

## 觀念確認（小任務）

你執行 `systemctl restart nginx` ，終端機沒有跳出任何錯誤訊息。這代表服務已經正常運作了嗎？

- A. 是，沒報錯就是成功
- B. 不一定，要再用 `status` 確認
- C. 要重開機才知道

**建議答案是 B。** `systemctl` 只是「接受了你的指令」。服務可能在背景嘗試啟動、失敗好幾次之後才放棄。**指令沒報錯，不等於服務活著。**
