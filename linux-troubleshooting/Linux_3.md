# Stage 3｜路徑一：設定寫錯
<br>

## 這一關的處境
<br>

`nginx -t` 已經指出來了：設定檔某一行少了一個分號。

原因找到了，現在要動手改。但前輩攔了你一下：

> 你等一下要改的，是正在服務全公司的設定檔。
>
> 改壞了，不是「再試一次」那麼簡單。動手前先想清楚三件事：改壞了能不能還原？你怎麼確定改對了？重啟的時候，會不會把正在用系統的人踢下線？

---
<br>

## 你今天只需要知道這些
<br>

改設定檔有一套固定順序，每一步都在防一件具體的事。

```text
備份 ─▶ 修改 ─▶ 驗證語法 ─▶ 套用 ─▶ 確認狀態 ─▶ 確認開機自動啟動
  │              │           │        │            │
  防「改壞了      防「帶著錯誤   讓新    防「以為修好    防「這次救活了
  回不去」        去重啟」      設定    了其實沒有」    下次開機又不見」
                              生效
```

備份不是為了讓你記得自己改了什麼，而是為了**在你不記得的時候還能還原**。

驗證語法這一步最容易被跳過，但它的價值最高：很多服務（包括 nginx）都提供語法檢查工具，**讓你在重啟之前就知道會不會失敗**。

### 套用設定有兩種方式

| 指令 | 它做什麼 | 對使用者的影響 |
| :--- | :--- | :--- |
| **reload** | 讓服務重新讀一次設定檔，程式本身不換 | 不斷線，使用者無感 |
| **restart** | 結束舊的程式，重新啟動一個新的 | 服務中斷幾秒，連線被切斷 |

正式環境改設定，優先用 `reload` 。

**但這次不行。** 服務現在是 failed，它根本沒在執行，沒有東西可以 reload。所以這次要用 `restart` 把它重新啟動。等它恢復正常之後，以後改設定就用 `reload` 。

## 角色 / 工具速查表
<br>

| 你想做 | 打這個 |
| :--- | :--- |
| 備份設定檔 | `sudo cp 原檔 原檔.bak` |
| 編輯設定檔 | `sudo nano 檔案` |
| 檢查 nginx 語法 | `nginx -t` |
| 套用設定（不斷線） | `sudo systemctl reload nginx` |
| 重新啟動服務 | `sudo systemctl restart nginx` |
| 確認狀態 | `systemctl status nginx` |
| 確認開機會自動啟動 | `systemctl is-enabled nginx` |
| 從備份還原 | `sudo cp 原檔.bak 原檔` |

> **nano 只需要記兩個鍵：** `Ctrl + O` 存檔（會問檔名，按 Enter 確認）、`Ctrl + X` 離開。
>
> 畫面最下面那排提示裡的 `^` 符號就是 Ctrl。

## 跟著做
<br>

**第一步：備份。這一步不能跳。**

`sudo cp /etc/nginx/sites-enabled/default /etc/nginx/sites-enabled/default.bak`{{execute}}

*→ 沒有任何輸出，就是成功了。*

**第二步：修改。找到剛才 `nginx -t` 指出的那一行，把漏掉的分號補上。**

`sudo nano /etc/nginx/sites-enabled/default`{{execute}}

*→ 改完按 Ctrl + O，Enter 確認，再按 Ctrl + X 離開。*

**第三步：驗證語法。重啟之前先確認。**

`sudo nginx -t`{{execute}}

*→ 你要看到 `syntax is ok` 和 `test is successful` 。*

> 如果還是報錯，**不要重啟**。回去 nano 再改，直到這一步通過為止。

**第四步：套用。服務沒在執行，所以用 restart 重新啟動。**

`sudo systemctl restart nginx`{{execute}}

*→ 一樣沒有輸出。但這不代表成功。*

**第五步：確認狀態。**

`systemctl status nginx`{{execute}}

*→ 你要看到 `active (running)` 。按 `q` 離開。*

`curl localhost`{{execute}}

*→ 應該吐出一大段 HTML。這才真正證明它在提供服務。*

**第六步：確認開機會自動啟動。**

`systemctl is-enabled nginx`{{execute}}

*→ 要看到 `enabled` 。如果是 disabled，補上一句 `sudo systemctl enable nginx` 。*

## 看懂結果
<br>

### 1. `nginx -t` 的兩種結果

| 你看到 | 這句話的意思 | 下一步 |
| :--- | :--- | :--- |
| **syntax is ok + test is successful** | 設定檔沒問題 | 可以套用重啟 |
| **[emerg] + 檔名 + 行號** | 還有錯，而且它告訴你在第幾行 | 回去改那一行，**不要重啟** |

設定檔類的錯誤是最好處理的一類，因為錯誤訊息會直接給你檔案路徑和行號，不需要你猜。

### 2. 修好之後該看到什麼

| 你看到 | 代表 |
| :--- | :--- |
| **active (running)** | 服務恢復執行 |
| **Main PID: 1234 (nginx)** | 有一個正在執行的程序編號 |
| **curl 吐出 HTML** | 網站真的有回應 |
| **is-enabled 回 enabled** | 下次開機它會自動啟動 |

> **一個判斷技巧：** `restart` 之後，程序編號（PID）會換成一個新的號碼。
>
> 如果你懷疑「它到底有沒有真的重啟」，看 PID 有沒有變就知道。這比看 `active (running)` 可靠——一個沒重啟成功、還跑著舊設定的服務，狀態一樣顯示 running。

## 常見誤會
<br>

**「改完直接重啟，失敗再說。」**

正式環境不能這樣。帶著語法錯誤去重啟，服務會直接停止，而且是在你手上停的。先跑 `nginx -t` ，通過了再重啟。這一步花三秒鐘，換掉一場事故。

**「restart 沒報錯，所以修好了。」**

`systemctl` 只是接受了你的指令。服務可能在背景嘗試啟動、失敗幾次之後放棄。**任何操作後都要用 `status` 確認。**

**「救活了就收工。」**

`start` 管的是**現在**跑不跑，`enable` 管的是**開機時**自動跑不跑。這是兩個獨立的開關。手動救活的服務如果從來沒 enable 過，三個月後機器重開，它不會自己起來——而且那時候沒人記得這件事。

**「reload 和 restart 反正都是套用設定。」**

對線上使用者不是。restart 會中斷連線，reload 不會。**改設定優先用 reload。**

## 觀念確認（小任務）
<br>

你在正式環境改了 nginx 的設定檔，想讓新設定生效，而且不想讓正在使用系統的同事斷線。你應該：

- A. `systemctl restart nginx`
- B. `systemctl reload nginx`
- C. 重開機

**建議答案是 B。** reload 讓服務重新讀取設定檔，程式本身不換，使用者不會斷線。restart 會結束舊程式，連線跟著被切斷。

*（但要注意：如果服務已經是 failed，它沒在執行，沒東西可以 reload，這時就得用 restart。）*
