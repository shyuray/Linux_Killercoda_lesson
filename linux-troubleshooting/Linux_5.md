# Stage 5｜路徑三：資源耗盡
<br>

## 這一關的處境
<br>

隔天早上，服務又停了。

你已經有分診的習慣，所以沒有慌。查 `status` ，看到 failed；讀執行紀錄，看到這一行：

```text
No space left on device
```

設定沒動過，權限也沒改過。前輩看了一眼：

> 這台機器不是壞了，是資源用完了。
>
> 服務不是只需要正確的設定和權限，它還需要磁碟空間、記憶體、CPU。任何一樣被用光，它都跑不起來。
>
> 而且，把磁碟塞滿的，常常就是執行紀錄本身。

---
<br>

## 你今天只需要知道這些
<br>

前兩條路徑，問題出在「你怎麼設定這台機器」。這一條不一樣，**問題出在這台機器的資源被用完了。**

三種資源耗盡，會導致三種不同的結果：

| 耗盡的資源 | 導致什麼 | 執行紀錄裡的線索 |
| :--- | :--- | :--- |
| **磁碟空間** | 服務寫不進東西，無法啟動 | `No space left on device` |
| **記憶體** | 系統為了自保，強制結束某個程式 | `Cannot allocate memory` 、 `Out of memory` |
| **CPU** | 服務不會停，但整台機器變得很慢 | 紀錄裡看不到，要用工具即時觀察 |

磁碟這一項有個常見的循環：服務每天不斷寫執行紀錄，如果沒人清理，幾個月後就把磁碟塞滿。這時候所有需要寫入的服務會一起停止。**第 2 關你拿來找原因的執行紀錄，在這一關變成了肇因。**

### 讓一個程式停下來，有兩種方式

有時候問題不是「東西壞了」，而是某個程式失控，把資源吃光、餓死其他人。這時候你要找出它、讓它停下來。

| 指令 | 它做什麼 | 什麼時候用 |
| :--- | :--- | :--- |
| `kill <PID>` | 送出結束訊號，讓程式有時間存檔、關閉檔案、清理乾淨再退出 | **預設先用這個** |
| `kill -9 <PID>` | 強制終止，程式沒有任何機會善後 | 前者無效時的最後手段 |

`kill -9` 會讓程式來不及寫完檔案、來不及釋放資源，可能留下損毀的資料。**它是最後手段，不是第一選擇。**

## 角色 / 工具速查表
<br>

| 你想知道 | 打這個 |
| :--- | :--- |
| 磁碟還剩多少空間？ | `df -h` |
| 是哪個資料夾在占用空間？ | `sudo du -sh /var/log/*` |
| 誰在吃 CPU 和記憶體？ | `top` （按 `q` 離開） |
| 記憶體剩多少？ | `free -h` |
| 某個程式的程序編號是多少？ | `ps aux \| grep 程式名` |
| 請一個程式結束 | `kill <PID>` |
| 強制終止（最後手段） | `kill -9 <PID>` |

> **兩個小提示：**
>
> `-h` 是 human-readable，把一長串位元組換成 1.2G、340M 這種看得懂的單位。
>
> `df` 看的是「整顆磁碟還剩多少」，`du` 看的是「某個資料夾用掉多少」。先用 `df` 確認滿了，再用 `du` 找出是誰占的。

## 跟著做
<br>

**第一步：確認磁碟真的滿了。**

`df -h`{{execute}}

*→ 看 `Use%` 那一欄。也注意 `Mounted on` ，它告訴你滿的是哪一個部分。*

**第二步：找出是誰在占用空間，由大到小排。**

`sudo du -sh /var/log/* | sort -rh`{{execute}}

*→ `sort -rh` 是「照人類可讀的大小、由大到小排序」。排在最上面的通常就是元凶。*

**第三步：確認元凶的大小。**

`ls -lh /var/log/nginx/error.log`{{execute}}

**第四步：清出空間。用清空，不要用刪除。**

`sudo truncate -s 0 /var/log/nginx/error.log`{{execute}}

*→ 把檔案內容清空，但檔案本身保留。服務可以繼續往裡面寫。*

**第五步：確認空間回來了。**

`df -h`{{execute}}

`ls -lh /var/log/nginx/error.log`{{execute}}

*→ 檔案還在，但大小變成 0。*

**第六步（另一種情況）：機器很慢，但服務沒停。**

`top`{{execute}}

*→ 看 `%CPU` 和 `%MEM` 兩欄，找出占用最高的那個程式，記下最左邊的程序編號。按 `q` 離開。*

先送出結束訊號（把 PID 換成你查到的號碼）：

`kill <PID>`{{copy}}

如果它沒有結束，再強制終止：

`kill -9 <PID>`{{copy}}

## 看懂結果
<br>

### 1. `df -h` 怎麼看

| 欄位 | 意思 |
| :--- | :--- |
| **Size** | 這個分割區總共多大 |
| **Used / Avail** | 用掉多少 / 還剩多少 |
| **Use%** | 百分比，**你的眼睛看這一欄** |
| **Mounted on** | 這個分割區對應到哪個路徑 |

使用率到 90% 就該處理，不要等到 100%。到 100% 的時候，很多服務已經停了，連你自己想寫一個暫存檔都可能失敗。

### 2. 為什麼用 truncate，不用 rm

```text
[原因] 服務還開著這個檔案
        │
        ▼  你用 rm 刪掉它
[結果] 檔名消失了，但磁碟空間沒有釋放
        │
        ▼  你會困惑：「明明刪了，為什麼還是滿的？」

正確做法：truncate -s 0
        把內容清空、保留檔案本身
        → 服務繼續寫，空間立刻釋放
```

### 3. `top` 怎麼看

| 看哪裡 | 找什麼 |
| :--- | :--- |
| **%CPU** | 誰的數字接近 100，甚至更高 |
| **%MEM** | 誰吃掉大部分記憶體 |
| **最左邊的 PID** | 記下來，等一下要用 |

`%CPU` 可以超過 100%，因為一個程式能同時使用多顆核心。看到 380% 不是故障，是它占用了將近四顆核心。

## 常見誤會
<br>

**「磁碟滿了，把那個大檔案直接 rm 掉就好。」**

服務還開著那個檔案的時候，rm 之後空間不會立刻釋放，而且服務會失去它的紀錄檔。**用 `truncate -s 0` 清空內容。**

**「服務停了，一定是設定或權限的問題。」**

這正是分診樹存在的理由。**設定對、權限對，磁碟滿了它照樣起不來。** 看到 `No space left on device` ，就不要再浪費時間去翻設定檔。

**「機器很慢，重開機就好。」**

重開之後，那個失控的程式可能又跟著啟動。你沒有解決問題，只是延後它發作。用 `top` 找出它，才知道要處理什麼。

**「kill 沒反應，那就一律用 kill -9。」**

`kill -9` 不給程式善後的機會，可能留下損毀的檔案或殘留的鎖定狀態。先用 `kill` ，等幾秒，確認沒有結束再用 `-9` 。

## 觀念確認（小任務）
<br>

`df -h` 顯示磁碟使用率 100%，服務因此無法啟動。你查出 `/var/log/nginx/error.log` 占了 8GB。最合適的處理方式是：

- A. `rm` 掉那個檔案，立刻釋放空間
- B. 用 `truncate -s 0` 清空檔案內容，保留檔案本身
- C. 重開機讓系統自己清理

**建議答案是 B。** 服務可能還開著那個檔案，直接 rm 掉，空間不一定會立刻釋放，而且服務會失去它的紀錄檔。清空內容、保留檔案，空間立刻回來，服務也能繼續寫入。
