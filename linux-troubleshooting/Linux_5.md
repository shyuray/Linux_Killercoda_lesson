# Stage 5｜路徑三：資源不足

## 這一關的處境
隔天早上，服務又掛了。
你已經有分診習慣了，所以你沒有慌。你查 `status`，看到 failed；你讀 log，看到這一行：

`No space left on device`

設定沒改過。權限也沒動過。
前輩看了一眼：

> 這台機器不是壞了，是撐不下去了。服務不是只需要正確的設定和權限，它還需要空間跑、需要記憶體用、需要 CPU 分給它。任何一樣被吃光，它就活不下去。
> 而且你要注意一件事：把磁碟塞爆的，常常就是 log 自己。

---

## 你今天只需要知道這些
前兩條路徑，問題出在「你怎麼設定這台機器」。這一條路徑不一樣——問題出在這台機器的家當被用完了。

**1. 資源崩潰的因果關係：**
* **因為**系統運行除了設定檔，還需要實體資源，**所以**當下列三種家當耗盡時，會引發不同的死法：
    * **磁碟空間滿了：** 服務無法寫入資料，**導致**啟動失敗（出現 `No space left on device`）。
    * **記憶體滿了：** 系統為了自保，**導致**程序被強制殺掉（出現 `Cannot allocate memory` 或 `Out of memory`）。
    * **CPU 滿了：** 服務不會掛掉，**但會導致**整台機器極度緩慢（log 看不到，必須用監控工具查）。

**2. 關於 log 檔案的因果：**
**因為**服務每天會不斷寫入日誌，**如果**沒人清理，**就會導致**磁碟在幾個月後被塞滿。你在第 2 關學會看的 log，現在變成殺手了。



**3. 關於終止程序（kill vs kill -9）的因果：**
有時候問題是某個程序失控吃光資源。你要找出它並讓它停下來。
* **`kill <PID>`（首選）：** **因為**這會先送出「禮貌請你結束」的訊號，讓程式有時間存檔、清乾淨再退出，**所以**最安全。
* **`kill -9 <PID>`（最後手段）：** **因為**這是直接從底層處決，程式來不及善後，**所以**極可能留下損毀的資料或檔案鎖。只有在前者無效時才用。

## 角色 / 工具速查表
| 你想知道 | 打這個 |
| :--- | :--- |
| 磁碟還剩多少空間？ | `df -h` |
| 是哪個資料夾在吃空間？ | `sudo du -sh /var/log/*` |
| 誰在吃 CPU 和記憶體？ | `top` (按 `q` 離開) |
| 某個程序的 PID 是多少？ | `ps aux \| grep 程序名` |
| 請一個程序結束 | `kill <PID>` |
| 強制殺掉（最後手段） | `kill -9 <PID>` |
| 記憶體剩多少？ | `free -h` |

> **💡 提示：** > * `-h` 代表 `human-readable`，把一長串數字換成 1.2G、340M 這種人看得懂的單位。
> * `df` 是看「整顆磁碟還剩多少」，`du` 是看「這個資料夾用掉多少」。先用 df 確認滿了，再用 du 找出誰在佔空間。

## 跟著做
請依序點擊或輸入以下指令，觀察右邊系統給你的反饋：

**第一步：確認磁碟真的滿了。**
`df -h`{{execute}}
*→ 看 `Use%` 那一欄。如果某一行是 100%，你找到問題了。注意 `Mounted on` 欄位，它告訴你滿掉的是哪個部分。*

**第二步：找出是誰在吃空間，並由大到小排序。**
`sudo du -sh /var/log/* | sort -rh`{{execute}}
*→ `sort -rh` 是「照人類可讀的大小，由大到小排」。最上面那個通常就是塞爆硬碟的元凶。*

**第三步：清出空間。**（這是一段示範指令，請修改為你查到的檔名）
`sudo truncate -s 0 /var/log/那個很大的檔`{{copy}}
*→ 找到肥大的 log 後，把檔案內容清空，但檔案本身保留。*

**第四步：確認空間回來了，並把服務救起。**
`df -h`{{execute}}
`sudo systemctl restart nginx`{{execute}}
`systemctl status nginx`{{execute}}

**第五步（另一種情境）：機器很慢，但服務沒掛。**
`top`{{execute}}
*→ 看 `%CPU` 和 `%MEM`，找出佔用最高的 PID（最左邊）。記下來後按 `q` 離開。*

然後先禮貌請它離開（請替換 PID）：
`kill <剛剛那個 PID>`{{copy}}
如果確認它死賴著不走，再處決它：
`kill -9 <剛剛那個 PID>`{{copy}}

## 看懂結果

**1. `df -h` 判讀因果：**
**因為**磁碟使用率到達 100% 時，連系統想寫個暫存檔都會失敗，**所以**當 `Use%` 看到 90% 時就該馬上處理，不要等到 100% 服務全死才救。



**2. 檔案清空的因果（為什麼用 truncate 而不是 rm？）：**
**因為**當服務還在運行並抓著該檔案時，直接 `rm` 刪除檔案，系統並不會立刻釋放磁碟空間（因為檔案把手沒放開），**所以**我們會遇到「明明刪了檔案，空間卻沒回來」的窘境。**因此**，正確做法是使用 `truncate -s 0` 把檔案「內容掏空，保留空殼」，這樣服務既能繼續寫入，磁碟空間也會立刻吐出來。

**3. `top` 判讀因果：**
**因為**一個程式可以同時呼叫多顆核心來運算，**所以**如果你在 `%CPU` 欄位看到超過 100%（例如 380%）的數字，這不是系統壞掉，而是它正在同時佔用將近四顆處理器核心。

## 常見誤會
* **「磁碟滿了，把那個大 log 直接 rm 掉就好。」**
    極度危險。如同上述因果，服務還抓著檔案時，`rm` 空間不會回來。請用 `truncate` 清空內容。
* **「服務掛了，一定是設定或權限的問題。」**
    這就是為什麼要有分診樹。設定對、權限對，磁碟滿了它照樣起不來。遇到 `No space left on device`，絕對不要浪費時間去檢查設定檔。
* **「機器很慢，重開機就好了。」**
    重開機後，那個失控的程序可能又跟著開機跑起來了。你沒解決問題，只是延後它發作。用 `top` 找出兇手，才知道要除掉什麼。
* **「kill 沒反應，那就一律 kill -9。」**
    `kill -9` 不給程序善後的機會，可能留下損毀的檔案或殘留的資料庫鎖。先用 `kill`，等幾秒，確認沒走再用 `-9`。

## 觀念確認（小任務）
`df -h` 顯示磁碟使用率 100%，服務因此無法啟動。你查出 `/var/log/nginx/error.log` 佔了 8GB。最合適的處理方式是：
* A. `rm` 掉那個檔案，立刻釋放空間
* B. 用 `truncate -s 0` 清空檔案內容，保留檔案本身
* C. 重開機讓系統自己清理

**建議答案是 B。** 如同剛才學到的因果關係，服務可能還開著那個檔案。直接 rm 掉，空間不一定會立刻釋放，而且服務會失去它的 log 檔。清空內容、保留檔案，空間立刻回來，服務也能無縫繼續寫入。
