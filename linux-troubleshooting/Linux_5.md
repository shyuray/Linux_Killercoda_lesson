# Stage 5｜路徑三：資源不足

## 這一關的處境

隔天早上，服務又掛了。

你已經有分診習慣了，所以你沒有慌。你查 `status` ，看到 failed；你讀 log，看到這一行：

```text
No space left on device
```

設定沒改過。權限也沒動過。

前輩看了一眼：

> 這台機器不是壞了，是 **撐不下去了** 。
>
> 服務不是只需要正確的設定和權限，它還需要空間跑、需要記憶體用、需要 CPU 分給它。任何一樣被吃光，它就活不下去。
>
> 而且你要注意一件事：**把磁碟塞爆的，常常就是 log 自己。**

---

## 你今天只需要知道這些

前兩條路徑，問題出在「你怎麼設定這台機器」。這一條路徑不一樣 —— **問題出在這台機器的家當被用完了。**

### 三種家當，三種死法

| 家當耗盡 | 導致的結果 | log 裡的線索 |
| :--- | :--- | :--- |
| **磁碟空間滿了** | 服務寫不進東西、無法啟動 | `No space left on device` |
| **記憶體滿了** | 系統為了自保，強制殺掉程序 | `Cannot allocate memory` 、 `Out of memory` |
| **CPU 滿了** | 服務不會掛，但整台機器極慢 | log 看不到，必須用工具即時查 |

> **一個殘酷的循環：**
>
> 服務每天不斷寫 log ─▶ 沒人清 ─▶ 幾個月後磁碟被塞滿 ─▶ 所有需要寫東西的服務一起死。
>
> 你在第 2 關學會看的 log，現在變成殺手了。

### 讓程序停下來的兩種方式

有時候問題不是「東西壞了」，而是**某個程序失控**，吃光資源把其他人餓死。這時你要找出它、並讓它停下來。

| 指令 | 做什麼 | 什麼時候用 |
| :--- | :--- | :--- |
| `kill <PID>` | **禮貌地請它結束** —— 它可以先存檔、關檔案、清乾淨再退出 | **預設，先試這個** |
| `kill -9 <PID>` | **直接處決** —— 程序沒有機會做任何善後 | 前者無效時，最後手段 |

> `kill -9` 會讓程序來不及寫完檔案、來不及釋放資源，可能留下損毀的資料或殘留的鎖。
>
> **它是最後手段，不是第一選擇。**

## 角色 / 工具速查表

| 你想知道 | 打這個 |
| :--- | :--- |
| 磁碟還剩多少空間？ | `df -h` |
| 是哪個資料夾在吃空間？ | `sudo du -sh /var/log/*` |
| 誰在吃 CPU 和記憶體？ | `top` （按 `q` 離開） |
| 記憶體剩多少？ | `free -h` |
| 某個程序的 PID 是多少？ | `ps aux \| grep 程序名` |
| 請一個程序結束 | `kill <PID>` |
| 強制殺掉（最後手段） | `kill -9 <PID>` |

> **兩個提示：**
>
> `-h` 代表 human-readable，把一長串數字換成 1.2G、340M 這種看得懂的單位。
>
> `df` 是看「**整顆磁碟**還剩多少」，`du` 是看「**這個資料夾**用掉多少」。先用 df 確認滿了，再用 du 找出誰在佔。

## 跟著做

**第一步：確認磁碟真的滿了。**

`df -h`{{execute}}

*→ 看 `Use%` 那一欄。注意 `Mounted on` 欄位，它告訴你滿掉的是哪個部分。*

**第二步：找出是誰在吃空間，由大到小排序。**

`sudo du -sh /var/log/* | sort -rh`{{execute}}

*→ `sort -rh` 是「照人類可讀的大小，由大到小排」。最上面那個，通常就是塞爆硬碟的元凶。*

**第三步：確認元凶的大小。**

`ls -lh /var/log/nginx/error.log`{{execute}}

*→ 這個 log 檔胖得不像話。*

**第四步：清出空間 —— 用清空，不要用刪除。**

`sudo truncate -s 0 /var/log/nginx/error.log`{{execute}}

*→ 把檔案內容掏空，但檔案本身保留。服務可以繼續往裡面寫。*

**第五步：確認空間回來了。**

`df -h`{{execute}}

`ls -lh /var/log/nginx/error.log`{{execute}}

*→ 檔案還在，但變成 0 bytes。*

**第六步（另一種情境）：機器很慢，但服務沒掛。**

`top`{{execute}}

*→ 看 `%CPU` 和 `%MEM` ，找出佔用最高的 PID（最左邊那欄）。記下來，按 `q` 離開。*

然後先禮貌請它離開（請替換成你查到的 PID）：

`kill <PID>`{{copy}}

如果確認它死賴著不走，再處決它：

`kill -9 <PID>`{{copy}}

## 看懂結果

### 1. `df -h` 怎麼讀

| 欄位 | 意思 |
| :--- | :--- |
| **Size** | 這個分割區總共多大 |
| **Used / Avail** | 用掉多少 / 還剩多少 |
| **Use%** | **百分比 —— 你的眼睛看這一欄** |
| **Mounted on** | 這個分割區對應到哪個路徑 |

> **Use% 到 90% 就該處理了，不要等到 100%。**
>
> 100% 的時候，很多服務已經死了，連你要寫個暫存檔都可能失敗。

### 2. 為什麼用 truncate 而不是 rm？

```text
[因] 服務還開著這個檔案的把手
      │
      ▼ 你 rm 掉檔案
[果] 檔案名字消失了，但磁碟空間沒有釋放
      │
      ▼ 你困惑：「明明刪了，為什麼還是滿的？」

✅ 正解：truncate -s 0
      把內容掏空、保留空殼
      → 服務繼續寫，空間立刻吐出來
```

### 3. `top` 怎麼讀

| 看哪裡 | 找什麼 |
| :--- | :--- |
| **%CPU 那一欄** | 誰的數字接近 100，甚至更高 |
| **%MEM 那一欄** | 誰吃掉大部分記憶體 |
| **最左邊 PID** | 記下來，等下要 kill 它 |

> **%CPU 可以超過 100%。**
>
> 因為一個程序可以同時用多顆核心。看到 380% 不是壞掉，是它佔了將近四顆核心。

## 常見誤會

**「磁碟滿了，把那個大 log 直接 rm 掉就好。」**

危險。服務還抓著檔案時，rm 之後空間不會立刻回來，而且服務會失去它的 log 檔。**用 `truncate -s 0` 清空內容。**

**「服務掛了，一定是設定或權限的問題。」**

這就是為什麼要有分診樹。**設定對、權限對，磁碟滿了它照樣起不來。** 遇到 `No space left on device` ，不要浪費時間去檢查設定檔。

**「機器很慢，重開機就好了。」**

重開之後那個失控的程序**可能又跑起來了**。你沒解決問題，只是延後它。用 `top` 找出兇手，才知道要處理什麼。

**「kill 沒反應，那就一律 kill -9。」**

`kill -9` 不給程序善後的機會，可能留下損毀的檔案或殘留的鎖。**先用 kill，等幾秒，確認沒走再用 -9。**

---

## 觀念確認（小任務）

`df -h` 顯示磁碟使用率 100%，服務因此無法啟動。你查出 `/var/log/nginx/error.log` 佔了 8GB。最合適的處理方式是：

- A. `rm` 掉那個檔案，立刻釋放空間
- B. 用 `truncate -s 0` 清空檔案內容，保留檔案本身
- C. 重開機讓系統自己清理

**建議答案是 B。** 服務可能還開著那個檔案的把手。直接 rm 掉，空間不一定會立刻釋放，而且服務會失去它的 log 檔。**清空內容、保留檔案**，空間立刻回來，服務也能無縫繼續寫入。
