## 電話一直響
<br>
<br>

上週你接手了公司的一台伺服器。你花了幾天摸清它的目錄結構、上面有哪些檔案、誰有權限碰什麼。

今天早上九點，**全公司在用的內部系統打不開了**。

人資、業務、工程部門全部卡住。電話一直響。

前輩沒有叫你重開機，也沒有叫你猜是誰改壞的。他只丟給你一句話：

> 先分類。你不知道它是哪一種壞法之前，任何動作都是亂試。

## 今天你要學會什麼

今天不用背完整本 Linux 百科全書。你只要跟著任務走，學會回答三個問題：

- 這個服務現在到底活著沒有？
- 它為什麼死掉？證據在哪裡？
- 救活之後，怎麼確定它不會再倒一次？

<br>
<br>

## 你先知道這個就好

**服務**，你可以先想成「這台機器上一直在待命的員工」。網站、資料庫、遠端連線，都是各自站崗的服務。

**systemd**，是「管所有員工上下班的總管家」。誰該上班、誰死了、開機時誰要自動來，都是它在管。

**systemctl**，是你手上的**對講機**。你不直接去搖那個員工，你跟總管家講話。

**log**，是每個員工的**工作日誌**。他做了什麼、出了什麼事，都寫在裡面。它是你的證據，不是裝飾。

維運不是「看到壞掉就修」，而是 **先分診，再處理**。

醫院急診不會一進門就開刀，會先分診：哪裡痛、多嚴重、送哪一科。伺服器一樣。

## 分診樹（今天最重要的一張圖）

面對「服務不正常」，你要依序回答三個問題。**由上往下，不要跳。**

```text
              「服務不正常」
                    │
   ┌────────────────┴────────────────┐
   │  步驟 1：它還活著嗎？             │
   └────────────────┬────────────────┘
                    │
      ┌─────────────┼─────────────┐
      ▼             ▼             ▼
 active(running) inactive(dead)  failed
   問題在別處      有人關掉它      往下走
                                   │
   ┌───────────────────────────────┘
   │
   ▼
┌─────────────────────────────────┐
│  步驟 2：它的遺言說什麼？（讀 log） │
└────────────────┬────────────────┘
                 │
   ┌─────────────┴─────────────┐
   │  步驟 3：遺言屬於哪一類？    │
   └─────────────┬─────────────┘
                 │
   ┌──────┬──────┴──────┬──────────┐
   ▼      ▼             ▼          ▼
 設定錯   權限錯      資源不足    Port 被佔
   │      │             │          │
 路徑一  路徑二        路徑三    （下一階段）
```

<br>
<br>
每一層都是下一層的前提 —— **不知道它死了沒，問「為什麼死」沒有意義。**

### 遺言關鍵字 → 對應路徑

| log 裡出現 | 屬於哪一類 |
| :--- | :--- |
| `syntax error`、`is not terminated by ";"` | 設定錯 → 路徑一 |
| `Permission denied`、`cannot open` | 權限錯 → 路徑二 |
| `No space left on device`、`Out of memory` | 資源不足 → 路徑三 |
| `Address already in use` | Port 被佔（下一階段教） |

## 任務地圖

```text
週一早上，內部系統打不開
  │
  ▼
Stage 1  確認災情：它還活著嗎？
  │
  ▼
Stage 2  找出死因：證據在 log 裡
  │
  ▼
Stage 3  路徑一：設定錯 —— 改設定、安全重啟
  │
  ▼
Stage 4  路徑二：權限錯 —— 服務不是用你的帳號跑的
  │
  ▼
Stage 5  路徑三：資源不足 —— 磁碟、記憶體、失控程序
  │
  ▼
Stage 6  追查真相：是誰在敲我們的門？
  │
  ▼
Stage 7  收尾：版本就是攻擊面
```

## 遊戲規則

1. **先看證據，再下結論。** 不要猜。
2. **只相信指令查到的現場狀態**，不相信「應該沒問題吧」。
3. **看到錯誤訊息不要慌。** 它通常已經告訴你兇手是誰。
4. **做完任何操作，再確認一次。** 沒報錯不等於做對了。

`whoami`{{execute}}

`hostname`{{execute}}

`uptime`{{execute}}

> **提示：** `uptime` 那個數字有用。如果它顯示「開機 5 分鐘」，代表這台機器剛剛重開過 —— 那本身就是一條線索。

## 觀念確認（小任務）

公司的內部系統打不開了，你的第一步應該是：

- A. 重開機
- B. 查看那個服務現在的狀態
- C. 猜是不是網路壞了

**建議答案是 B。** 分診的第一步是「它還活著嗎」。重開機會摧毀證據，猜測會讓你查錯方向。
