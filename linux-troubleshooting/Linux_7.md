> 密碼只是一道門。如果你機器上跑的軟體是三年前的舊版，而那個版本有公開的漏洞，攻擊者根本不需要猜密碼。

所以最後一件事：去確認我們裝了什麼、版本多舊、有沒有該更新的。

## 你今天只需要知道這些
這台機器上裝的每一個軟體，都是一個可能被攻擊的入口。軟體越舊，已知漏洞越多。所以「這台機器裝了什麼、版本是多少」不是維運瑣事，它是資安工作的第一份清單。

Linux 用套件管理系統（Ubuntu 的工具叫 `apt`）來管這件事。
它幫你解決的最大麻煩是「相依性」：你要裝 nginx，nginx 需要某些底層函式庫，函式庫又需要別的東西。`apt` 會自動把這一整串連鎖裝好。不要去網路上隨便抓安裝檔手動裝，你會漏掉相依套件，或裝到來源不明的版本。

**有兩個極度容易搞混的分辨（因果關係）：**



**1. `update` 不是「更新軟體」**
| 指令 | 實際做什麼（因） | 結果（果） |
| :--- | :--- | :--- |
| **`apt update`** | 去官方倉庫拿一份最新的「目錄」 | 你手上的軟體一個都沒變 |
| **`apt upgrade`** | 照著剛剛拿到的目錄，去下載並安裝新版 | 軟體真的升級了 |
*→ **所以**順序永遠是先 update 再 upgrade。反過來，你會照著一份過期的舊目錄去升級。*

**2. `remove` 不是「清乾淨」**
| 指令 | 實際做什麼（因） | 資安上的影響（果） |
| :--- | :--- | :--- |
| **`apt remove`** | 移除程式執行檔，**但保留設定檔** | 如果設定檔被塞了後門，威脅依然躺在 `/etc` 裡 |
| **`apt purge`** | 移除程式，**連設定檔一起刪** | 徹底連根拔起，不留痕跡 |

## 角色 / 工具速查表

| 你想知道 | 打這個 |
| :--- | :--- |
| 這台機器裝了哪些套件？ | `apt list --installed` |
| 某個套件裝了沒？版本多少？ | `dpkg -l \| grep nginx` |
| 這個套件的版本與來源？ | `apt policy nginx` |
| 更新目錄，然後升級 | `sudo apt update && sudo apt upgrade` |
| 移除（留設定檔） | `sudo apt remove 套件名` |
| 移除（連設定檔一起清） | `sudo apt purge 套件名` |
| 這個指令的執行檔在哪？ | `which nginx` |

> **💡 提示：** `apt policy` 會告訴你目前裝的是哪一版、倉庫裡有沒有更新的版本、以及這個套件是從哪個來源來的。查漏洞、確認來源可信，都看這個。

## 跟著做
請依序點擊或輸入以下指令，觀察右邊系統給你的反饋：

**第一步：確認我們裝的 nginx 是什麼版本。**
`apt policy nginx`{{execute}}
*→ 看兩行：已安裝 (Installed) 和 候選 (Candidate)。*

`dpkg -l | grep nginx`{{execute}}
*→ 開頭的 `ii` 代表「已安裝且設定完成」。後面跟著的就是版本號。*

**第二步：更新目錄，看看有什麼可以升級。**
`sudo apt update`{{execute}}
*→ 這一步不會動到任何軟體，只是去拿最新的目錄。最後幾行會告訴你「有幾個套件可以升級」。*

`apt list --upgradable`{{execute}}
*→ 列出所有可以升級的套件，以及它們的新舊版本。這張清單就是你的待辦事項。*

**第三步：升級。**
`sudo apt upgrade`{{execute}}
*→ 它會列出要動哪些套件、下載多大，並問你 `Y/n`。輸入 `Y` 按 Enter 繼續。*

**第四步：確認服務還活著（最重要的一步）。**
`systemctl status nginx`{{execute}}
`curl localhost`{{execute}}
*→ 升級可能會重啟服務。你做完任何維護動作，都要回頭確認服務狀態。*

**第五步：查一個指令的執行檔在哪（排查時常用）。**
`which nginx`{{execute}}

## 看懂結果

**1. `apt policy` 的因果判讀：**
```text
nginx:
  Installed: 1.24.0-1ubuntu1     ← 你現在裝的版本
  Candidate: 1.24.0-1ubuntu1.2   ← 倉庫裡最新的版本
```
| 情況 | 代表的意思 |
| :--- | :--- |
| **Installed = Candidate** | 已經是最新的，很安全 |
| **Installed 舊於 Candidate** | **因為**有更新沒裝，**所以**可能有已知漏洞暴露在外 |
| **Installed: (none)** | 這台機器根本沒裝這個套件 |

**2. `dpkg -l` 開頭那兩個字母的意義：**
| 你看到 | 代表 | 稽核線索 |
| :--- | :--- | :--- |
| **ii** | 已安裝、設定完成 | 正常運作中 |
| **rc** | 程式已移除，但設定檔還在 | **因為**有人只用了 remove，**所以**那個殘留的設定檔值得看一眼 |

## 常見誤會
* **「apt update 就是更新軟體。」**
    不是。它只更新目錄。真正升級軟體的是 upgrade。這是最常被搞混的一組。
* **「反正都是移除，remove 就夠了。」**
    remove 會留下設定檔。如果你是因為懷疑某個套件不安全才移除它，設定檔留著就沒有清乾淨。這種時候要用 purge。
* **「一次把所有東西升到最新，最安全。」**
    安全，但不一定穩定。升級可能重啟服務、改變設定檔、甚至讓相依的東西壞掉。正式環境要先確認影響，不要在上班時間 upgrade 完就走人。升完一定要 `systemctl status` 確認服務還活著。
* **「這台機器我沒裝什麼東西，應該很安全。」**
    系統預設就裝了幾百個套件。你沒裝的東西，一樣會有漏洞。這就是為什麼要定期 update && upgrade，而不是只管自己裝的那幾個。

## 觀念確認（小任務）
你懷疑某個套件的設定檔被植入了惡意設定，想把它徹底從系統上清除。你應該：
* A. `sudo apt remove 套件名`
* B. `sudo apt purge 套件名`
* C. `sudo rm 掉那個程式的執行檔`

**建議答案是 B。**
因為 remove 只移除程式本體，設定檔會留在系統裡。purge 才會連設定檔一起清除。至於 C，手動 rm 執行檔會讓套件系統的紀錄和實際狀態不一致，以後排查更麻煩，絕對不要這樣做。

## 🎯 這一階段你帶走了什麼？

指令會忘，但這幾件事不會：

1. **一張分診樹：** 面對「服務不正常」，你不再亂試。先確認它活著沒 → 讀它的遺言 → 判斷是設定、權限、還是資源 → 走對應的路。這套方法能處理你沒見過的故障，這才是它的價值。
2. **四個致命判斷：**
    * **沒報錯不等於做對了：** restart 不報錯，服務可能還躺著。任何操作後都要確認。
    * **「你」讀得到，不代表「它」讀得到：** 服務用自己的低權限帳號跑。判斷權限時，永遠先問「是誰要去讀」。
    * **改設定之前先備份，重啟之前先驗語法：** 三秒鐘的動作，省你一場事故。
    * **讓資料說話：** 你憑印象猜的地方，就是攻擊者藏身的地方。
3. **一條可以複製的管線：** `grep | awk | sort | uniq -c | sort -rn`。換掉關鍵字和欄位，它能回答任何「誰出現最多次」的資安與維運問題。
