# Stage 4｜路徑二：權限錯
<br>

## 這一關的處境

服務活著。`systemctl status` 顯示 `active (running)` ，`curl localhost` 也吐得出首頁。

但同事說，點「報表下載」那一頁就跳 **403 Forbidden** 。

你自己上去看那個檔案 —— 用 `cat` 讀，讀得出來，內容好好的在那裡。

你回報前輩：「檔案沒問題啊，我讀得到。」

前輩問你一句：

> 「**你**」讀得到，不代表「**它**」讀得到。
>
> 你知道 nginx 是用哪個帳號在跑的嗎？

---
<br>

## 你今天只需要知道這些

這是整個階段最反直覺、也最常在現場出事的一件事：

### **服務不是用你的帳號在跑。**

你登入這台機器是 `ubuntu` ，你在 sudo 群組裡，權力很大。

但 nginx 這個服務，系統是用一個叫 `www-data` 的**專屬低權限帳號**在跑它。那個帳號不能 sudo、不能登入，只能碰它該碰的東西。

**為什麼這樣設計？** 這叫**最小權限原則**：萬一 nginx 被攻破，攻擊者拿到的只是 `www-data` 這個小權限帳號，不能變成 root。

> **所以：** 系統判斷「誰能碰這個檔」時，身為外人的服務帳號只會吃到 **others（其他人）** 那一格的權限。

### 還有一個更陰的坑：資料夾的門

要讀取檔案，必須先能「**走進**」它所在的資料夾 —— 那需要 `x` 權限。

```text
[因] /var/www/html/reports/     ← 資料夾：others 沒有 x（門是鎖的）
          │
          ▼ 所以 www-data 走不進去
[果]      └── report.txt        ← 檔案：即使設為 644 人人可讀
                                   依然被擋在門外
```

**檔案設得再開放都沒用，因為門是鎖的。** 這就是新手最崩潰的「我明明有權限為什麼還是 denied」。

今天只要建立這兩個判斷：

1. **服務用別的帳號跑。**
2. **父資料夾的 `x` 會擋死一切。**

---
<br>

## 角色 / 工具速查表

| 你想知道 | 打這個 |
| :--- | :--- |
| 這個服務是用哪個帳號在跑？ | `ps aux \| grep nginx` |
| 這個檔案的權限與主人 | `ls -l 檔案` |
| **這個資料夾本身**的權限 | `ls -ld 資料夾` |
| nginx 遇到什麼錯？ | `sudo tail -n 5 /var/log/nginx/error.log` |
| 改主人 | `sudo chown www-data:www-data 檔案` |
| 給資料夾「可進入」的權限 | `sudo chmod o+x 資料夾` |

> **關鍵提示：** `ls -l` 和 `ls -ld` 差一個字母，差很多。
>
> `ls -l 資料夾` 是「列出資料夾**裡面**有什麼」。
>
> `ls -ld 資料夾` 是「看資料夾**本身**的權限」。
>
> 診斷這類問題，你要看的是後者。

---
<br>

## 跟著做

**第一步：先確認服務是用誰的身分在跑。**

`ps aux | grep nginx`{{execute}}

*→ 你會看到兩種程序：root 跑的 master，和 `www-data` 跑的 worker。真正處理網頁請求、去讀檔案的身分是 `www-data` 。*

**第二步：看 nginx 自己怎麼說。它有專屬的錯誤日誌。**

`sudo tail -n 5 /var/log/nginx/error.log`{{execute}}

*→ 找 `Permission denied` 這幾個字，以及它說碰不到哪個路徑。*

**第三步：檢查那個檔案。**

`ls -l /var/www/html/reports/report.txt`{{execute}}

*→ 看 others 那三格。它是 `r--` 還是 `---` ？*

**第四步：檢查那個資料夾本身。這步最容易被跳過。**

`ls -ld /var/www/html/reports`{{execute}}

*→ 看 others 那三格有沒有 `x` 。沒有 `x` ，www-data 就走不進去。*

**第五步：修。把資料夾的門打開。**

`sudo chmod o+x /var/www/html/reports`{{execute}}

**第六步：驗證。**

`curl localhost/reports/report.txt`{{execute}}

*→ 讀得到內容，就修好了。*

> **注意：這一關不需要 restart nginx。**
>
> 你改的是檔案系統的權限，不是 nginx 的設定。服務會在下一次請求時直接讀到新的權限。
>
> 不要養成「什麼都 restart 一下」的習慣 —— 那會不必要地中斷服務。

## 看懂結果

### 1. 找出真正在做事的帳號

`ps aux` 的第一欄，就是執行該程序的帳號：

```text
root      1234  ... nginx: master process /usr/sbin/nginx
www-data  1235  ... nginx: worker process     ← 真正讀檔案的是它！
ubuntu    1240  ... grep --color=auto nginx   ← 這是你自己的搜尋指令
```

### 2. 錯誤訊息的因果判讀

| 你看到 | 根本原因（因） | 下一步（果） |
| :--- | :--- | :--- |
| **Permission denied + 路徑** | 服務帳號碰不到該檔或路徑 | 用 `ls -l` 看檔、`ls -ld` 看資料夾 |
| **403 Forbidden** | 伺服器讀不到檔，拒絕吐資料 | 同上，排查權限 |
| **404 Not Found** | 檔案「真的不存在」 | 去查路徑是不是打錯，這不是權限問題 |

> **403 和 404 要分清楚：**
>
> **403** 是「東西在，但不給你」。
>
> **404** 是「東西根本不在」。
>
> 兩者的排查方向完全不同。

## 常見誤會

**「我 cat 讀得到，所以檔案權限沒問題。」**

你是 ubuntu，而且你可以 sudo。服務是 www-data，它的權限和你完全不同。**判斷權限的時候，永遠要問「是誰要去讀」。**

**「檔案設 644 人人可讀，一定讀得到。」**

如果它所在的資料夾對 others 沒有 `x` ，服務連走進去都不行，檔案設什麼都沒用。**檢查檔案之後，一定要 `ls -ld` 檢查它的父資料夾。**

**「權限有問題，那就 chmod 777 全開，一定能解決。」**

會解決，也會製造一個資安漏洞。777 代表**這台機器上任何人都能改寫這個檔**。萬一有其他服務被攻破，攻擊者就能改你的網頁內容。**給剛好夠用的權限，不要圖方便全開。**

**「權限改完要重啟服務才生效。」**

不用。檔案權限是作業系統層的，服務下次讀取時就吃到新的了。**能不重啟就不重啟。**

## 觀念確認（小任務）

一個檔案的權限是 644（擁有者可讀寫，其他人唯讀），但 nginx 服務仍然回報 `Permission denied` 。最可能的原因是：

- A. 檔案權限不夠，要改成 777
- B. 檔案所在的資料夾，沒有給 others `x` 權限，服務走不進去
- C. nginx 需要重新啟動

**建議答案是 B。** 檔案本身人人可讀沒錯，但要碰到它，得先能進入它所在的資料夾。**父資料夾的 `x` 權限會蓋過檔案的權限。** 這是排查權限問題時最容易被忽略的一層。
