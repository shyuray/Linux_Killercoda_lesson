# Stage 4｜路徑二：權限不足
<br>

## 這一關的處境
<br>

服務在執行。`systemctl status` 顯示 active (running)，`curl localhost` 也吐得出首頁。

但同事說，點「報表下載」那一頁就跳出 403 Forbidden。

你自己上去看那個檔案，用 `cat` 讀得出來，內容好好的在那裡。你回報前輩：「檔案沒問題啊，我讀得到。」

前輩問你一句：

> 「你」讀得到，不代表「它」讀得到。
>
> 你知道 nginx 是用哪個帳號在跑的嗎？

---
<br>

## 你今天只需要知道這些
<br>

這是整個階段最反直覺、也最常在現場出事的一件事：**服務不是用你的帳號在執行。**

你登入這台機器的身分是 `ubuntu` ，你在 sudo 群組裡，權限很大。但 nginx 這個服務，系統是用一個叫 `www-data` 的專屬帳號在跑它，那個帳號權限很小，不能 sudo，甚至不能登入。

這樣設計是刻意的，叫做**最小權限原則**：萬一 nginx 被攻破，攻擊者拿到的只是 `www-data` 這個低權限帳號，不會直接變成整台機器的管理者。

於是問題來了。系統在判斷「誰能碰這個檔」的時候，是照 **擁有者 → 所屬群組 → 其他人** 的順序比對，中一個就停。`www-data` 既不是檔案的擁有者、也不在它的群組裡，所以它只會吃到 **其他人（others）** 那一格的權限。

### 還有一個更容易漏掉的地方

要讀到一個檔案，你必須先能「進入」它所在的資料夾。而「進入資料夾」這個動作，需要的是資料夾的 `x` 權限。

```text
[原因] /var/www/html/reports/     ← 資料夾對 others 沒有 x
            │
            ▼  www-data 進不去這個資料夾
[結果]      └── report.txt        ← 檔案就算設成人人可讀
                                     也一樣讀不到
```

**檔案權限設得再開放都沒用，因為外面那道門是鎖的。** 這就是新手最常卡住的「我明明有權限，為什麼還是 Permission denied」。

今天只要建立兩個判斷：**服務用的是別的帳號**，以及**父資料夾的 `x` 會蓋過檔案本身的權限**。

---
<br>

## 角色 / 工具速查表
<br>

| 你想知道 | 打這個 |
| :--- | :--- |
| 這個服務是用哪個帳號在跑？ | `ps aux \| grep nginx` |
| 這個檔案的權限與擁有者 | `ls -l 檔案` |
| **這個資料夾本身**的權限 | `ls -ld 資料夾` |
| nginx 遇到什麼錯？ | `sudo tail -n 5 /var/log/nginx/error.log` |
| 改擁有者 | `sudo chown www-data:www-data 檔案` |
| 讓 others 能進入資料夾 | `sudo chmod o+x 資料夾` |

> **`ls -l` 和 `ls -ld` 差一個字母，意思完全不同。**
>
> `ls -l 資料夾` 是列出資料夾**裡面**有什麼。
>
> `ls -ld 資料夾` 是看資料夾**自己**的權限。
>
> 排查這類問題，你要看的是後者。

---
<br>

## 跟著做
<br>

**第一步：先確認服務是用誰的身分在執行。**

`ps aux | grep nginx`{{execute}}

*→ 你會看到兩種程序：root 執行的 master，以及 `www-data` 執行的 worker。真正處理網頁請求、去讀檔案的是 worker，也就是 `www-data` 。*

**第二步：看 nginx 自己怎麼說。它有專屬的錯誤紀錄。**

`sudo tail -n 5 /var/log/nginx/error.log`{{execute}}

*→ 找 `Permission denied` 這幾個字，以及它說碰不到哪個路徑。*

**第三步：檢查那個檔案。**

`ls -l /var/www/html/reports/report.txt`{{execute}}

*→ 看 others 那三格，是 `r--` 還是 `---` ？*

**第四步：檢查那個資料夾本身。這一步最容易被跳過。**

`ls -ld /var/www/html/reports`{{execute}}

*→ 看 others 那三格有沒有 `x` 。沒有 `x` ，`www-data` 就進不去。*

**第五步：把資料夾的門打開。**

`sudo chmod o+x /var/www/html/reports`{{execute}}

**第六步：驗證。**

`curl localhost/reports/report.txt`{{execute}}

*→ 讀得到內容，就修好了。*

> **注意：這一關不需要 restart nginx。**
>
> 你改的是檔案系統的權限，不是 nginx 的設定。服務會在下一次請求時直接讀到新的權限。
>
> 不要養成「什麼都 restart 一下」的習慣，那會不必要地中斷服務。

## 看懂結果
<br>

### 1. `ps aux` 的第一欄，就是執行這支程式的帳號

```text
root      1234  ... nginx: master process /usr/sbin/nginx
www-data  1235  ... nginx: worker process     ← 真正去讀檔案的是它
ubuntu    1240  ... grep --color=auto nginx   ← 這是你自己的搜尋指令
```

### 2. 三種錯誤訊息，指向不同方向

| 你看到 | 原因 | 該查哪裡 |
| :--- | :--- | :--- |
| **Permission denied + 路徑** | 服務帳號碰不到那個檔或路徑 | `ls -l` 看檔案、`ls -ld` 看資料夾 |
| **403 Forbidden** | 伺服器讀不到檔案，拒絕回應 | 同上，排查權限 |
| **404 Not Found** | 檔案真的不存在 | 檢查路徑有沒有打錯，這不是權限問題 |

403 和 404 要分清楚：**403 是「東西在，但不給你」，404 是「東西根本不在」。** 排查方向完全不同。

## 常見誤會
<br>

**「我 cat 讀得到，所以檔案權限沒問題。」**

你是 ubuntu，而且可以 sudo。服務是 www-data，權限跟你完全不同。**判斷權限的時候，永遠要先問「是誰要去讀」。**

**「檔案設成 644 人人可讀，一定讀得到。」**

如果它所在的資料夾對 others 沒有 `x` ，服務連進去都不行，檔案設什麼都沒用。**查完檔案，一定要再用 `ls -ld` 查它的父資料夾。**

**「權限有問題，那就 chmod 777 全開，一定能解決。」**

能解決，也會製造一個資安漏洞。777 代表這台機器上任何人都能改寫這個檔案。萬一有別的服務被攻破，攻擊者就能改你的網頁內容。**給剛好夠用的權限，不要圖方便全開。**

**「權限改完要重啟服務才生效。」**

不用。檔案權限是作業系統層的設定，服務下次讀取時就會吃到新的。**能不重啟就不重啟。**

## 觀念確認（小任務）
<br>

一個檔案的權限是 644（擁有者可讀寫，其他人唯讀），但 nginx 仍然回報 Permission denied。最可能的原因是：

- A. 檔案權限不夠，要改成 777
- B. 檔案所在的資料夾沒有給 others `x` 權限，服務進不去
- C. nginx 需要重新啟動

**建議答案是 B。** 檔案本身人人可讀沒錯，但要碰到它，得先能進入它所在的資料夾。**父資料夾的 `x` 權限會蓋過檔案本身的權限**，這是排查權限問題時最容易被忽略的一層。
