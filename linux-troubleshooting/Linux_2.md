# Stage 2｜從執行紀錄裡找出原因
<br>

## 這一關的處境
<br>

狀態是 failed。不是有人把它關掉，是它自己啟動時出了錯。

前輩說：

> 服務停掉不會不吭聲，它在停之前一定寫了東西。
>
> 你的工作不是猜它為什麼停，是把它寫的東西找出來讀懂。

---
<br>

## 你今天只需要知道這些
<br>

服務在停止的那一刻，通常會把「我遇到了什麼問題」寫進執行紀錄。**所以你不需要推理，只需要找到那句話。**

執行紀錄分成兩個地方存放，讀它們的工具不同，不要拿錯：

| 存放位置 | 內容 | 用什麼讀 |
| :--- | :--- | :--- |
| **systemd 統一收集** | 各個服務的啟動、停止、錯誤 | `journalctl` |
| **`/var/log/` 底下的檔案** | 系統與各程式自己寫的紀錄 | `cat` 、 `less` 、 `tail` 、 `grep` |

服務為什麼停，去第一個地方找。誰登入過這台機器，去第二個地方找（第 6 關會用到）。

還有一件事能幫你省下大量時間：**執行紀錄裡絕大部分是雜訊。** 一個服務每天寫幾千行，多數是「我正常運作中」的例行訊息。你要學的不是讀得快，而是懂得過濾——**只撈出錯誤等級的那幾行。**

---
<br>

## 角色 / 工具速查表
<br>

`journalctl` 有數十個參數，今天只會用到四個。

| 你想知道 | 打這個 |
| :--- | :--- |
| 這個服務最近說了什麼？ | `journalctl -u nginx -n 20` |
| 只要錯誤，不要雜訊 | `journalctl -u nginx -p err` |
| 只看最近十分鐘的紀錄 | `journalctl -u nginx --since "10 min ago"` |
| 一邊操作一邊即時觀察它 | `journalctl -u nginx -f` |
| 上週的舊紀錄呢？ | `zgrep "關鍵字" /var/log/syslog.2.gz` |

> **參數說明：** `-u` 指定服務、`-p err` 只顯示錯誤等級、`-n 20` 只看最後 20 行、`-f` 即時跟著新進的紀錄滾動。

> **關於那些 .gz 檔：** 執行紀錄會無限長大，所以系統會定期把舊的封存並壓縮，這個機制叫 log rotation，結果就是 `syslog.1` 、 `syslog.2.gz` 這些檔案。
>
> 要查上週發生的事，就得翻這些舊檔。壓縮過的檔案不能直接用 `grep` ，要改用 `zgrep` 。這是新手第一次追查歷史事故時最容易卡住的地方。

## 跟著做
<br>

**第一步：先看它最後說了什麼。**

`journalctl -u nginx -n 20`{{execute}}

*→ 你會看到大量重複的訊息，因為 systemd 反覆嘗試啟動、反覆失敗。先別急著讀，下一個指令會幫你濾掉雜訊。*

**第二步：只留下錯誤等級的行。**

`journalctl -u nginx -p err`{{execute}}

*→ 資訊量會少掉九成。原因就在剩下的這幾行裡。*

**第三步：讓 nginx 自己檢查設定檔。**

`nginx -t`{{execute}}

*→ 這是 nginx 內建的語法檢查，它會直接告訴你：哪個檔案、第幾行、錯在哪裡。*

> 記下 `nginx -t` 給你的檔名和行號，下一關要用。

## 看懂結果
<br>

### 1. 執行紀錄有等級，先掃後面兩種

| 等級 | 意思 | 該不該理它 |
| :--- | :--- | :--- |
| **INFO / Started** | 例行訊息：「我開始工作了」 | 雜訊，跳過 |
| **WARNING** | 有點異常，但還能運作 | 記著，不急 |
| **ERROR / Failed** | 出錯了 | **這才是你要找的** |

### 2. 錯誤訊息之間有因果，真正的原因在最上面

```text
[原因] nginx: [emerg] directive is not terminated by ";"
                      in /etc/nginx/sites-enabled/default:N
        │
        ▼  設定檔語法錯，啟動前的檢查沒過
[結果] nginx.service: Control process exited, code=exited, status=1
        │
        ▼  檢查沒過，服務啟動失敗
[結果] Failed to start nginx.service
```

新手常常盯著最後一行的 `Failed to start` 發呆。那行只告訴你「它失敗了」，不會告訴你「為什麼」。**真正的原因在更上面，往上找第一個 error。**

### 3. 錯誤訊息對應的分診路線

| 執行紀錄裡出現 | 屬於哪一類 | 下一步 |
| :--- | :--- | :--- |
| `syntax error` 、 `is not terminated by ";"` | 設定寫錯 | 路徑一 |
| `Permission denied` 、 `cannot open` | 權限不足 | 路徑二 |
| `No space left on device` 、 `Out of memory` | 資源耗盡 | 路徑三 |
| `Address already in use` | 連接埠被占用 | 下一階段 |

## 常見誤會
<br>

**「紀錄太多了，根本讀不完。」**

不用讀完。用 `-p err` 只撈錯誤，通常只剩三五行。**讀 log 的能力不是讀得快，而是知道怎麼過濾。**

**「有顏色的、紅色的才是錯誤。」**

顏色只是終端機幫你加上的輔助。當你把紀錄導到檔案、或用管線處理時，顏色會完全消失。你要練的是讀懂文字，不是找顏色。

**「最後一行寫 Failed，那就是原因。」**

最後一行通常是結果，不是原因。原因在更上面。

## 觀念確認（小任務）
<br>

執行紀錄裡有這兩行：

```text
Unable to locate executable '/usr/sbin/nginxXXX': No such file or directory
Main process exited, code=exited, status=203/EXEC
```

哪一行是根本原因？

- A. 第一行——找不到執行檔
- B. 第二行——程式結束，代碼 203
- C. 兩行都是原因

**建議答案是 A。** 因為找不到那個執行檔，程式才無法啟動，才回報 203。第二行是第一行造成的結果。你修得了第一行，修不了第二行。
