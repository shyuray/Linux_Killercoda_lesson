# Stage 2｜它的遺言說什麼？

## 這一關的處境
`Active: failed`。它不是被人關掉的，是自己死的。
前輩說：

> 服務死掉不會不吭聲。它在倒下之前，一定寫了東西。你的工作不是猜它為什麼死，是去把它寫的東西找出來讀懂。

---

## 你今天只需要知道這些
**log（日誌），**你可以先想成每個服務的工作日誌。它做了什麼、遇到什麼問題、幾點幾分發生的，全都寫在裡面。

服務死掉的那一刻，它通常會把「我為什麼活不下去」寫進去。**因為服務在崩潰前必定會將最後遭遇的致命錯誤印出來，所以只要循著這些紀錄，就能精準反推回發問題的源頭。** log 不是給你參考的補充資料，它就是命案現場的鐵證，你不需要推理誰是兇手，你只要找到那句自白。

Linux 的 log 分成兩個世界，你要分得清楚，因為讀它們的工具完全不同：

| 世界 | 東西在哪 | 用什麼讀 |
| :--- | :--- | :--- |
| **systemd 管的服務日誌** | 由 systemd 統一收著，不是普通檔案 | `journalctl` |
| **傳統文字檔** | `/var/log/` 底下一堆檔 | `cat`、`less`、`tail`、`grep` |

> **💡 提示：** 服務為什麼掛，要去第一個世界找；誰登入過這台機器，要去第二個世界找（第 6 關會用到）。
> 另外，log 大部分是雜訊。一個服務每天寫幾千行，絕大多數是例行報告。你要學會只撈出「錯誤等級」的那幾行，而不是從頭讀到尾。

---

## 角色 / 工具速查表
`journalctl` 有幾十個參數，今天只用這四個：

| 你想知道 | 打這個 |
| :--- | :--- |
| 這個服務最近說了什麼？ | `journalctl -u nginx -n 20` |
| 只要錯誤，不要雜訊 | `journalctl -u nginx -p err` |
| 只看最近十分鐘的 | `journalctl -u nginx --since "10 min ago"` |
| 一邊操作一邊即時看它吐什麼 | `journalctl -u nginx -f` |
| 上週的舊 log 呢？ | `zgrep "關鍵字" /var/log/syslog.2.gz` |

> **💡 參數解析：** `-u` 是指定服務，`-p err` 是只看錯誤等級，`-n 20` 是只看最後 20 行，`-f` 是即時跟著滾。這四個組合起來就能應付大部分現場。
> 
> **關於舊檔案：** log 會無限長大，所以系統會定期把舊的封存壓縮（這叫 log rotation），變成 `syslog.1`、`syslog.2.gz`。要查上週的事，壓縮過的檔不能直接 `grep`，要用 `zgrep`。這是很多新手第一次查歷史事故時撞的牆。

## 跟著做
請依序點擊或輸入以下指令，觀察右邊系統給你的反饋：

`journalctl -u nginx -n 20`{{execute}}
*→ 最後 20 行。你會看到一堆重複的訊息——systemd 反覆嘗試啟動、反覆失敗。先別急著讀，下一個指令會幫你濾掉雜訊。*

`journalctl -u nginx -p err`{{execute}}
*→ 只剩錯誤等級的行。資訊量瞬間少了九成，兇手就在這幾行裡。*

`nginx -t`{{execute}}
*→ 這是 nginx 自己的設定檔語法檢查。它會直接告訴你：哪個檔案、第幾行、錯在哪裡。請記下這裡給的檔名和行號，下一關要用。*

## 看懂結果
log 有等級，你的眼睛要先掃後面兩種：

| 等級 | 意思 | 你該不該理它 |
| :--- | :--- | :--- |
| **INFO / Started** | 例行報告：「我開始工作了」 | 雜訊，跳過 |
| **WARNING** | 有點怪，但還能跑 | 記著，不急 |
| **ERROR / Failed** | 出事了 | 這就是你要找的錯誤訊息 |

**讀懂因果關係（最重要）：**
**系統記錄錯誤時，通常是「最底層的事件（因）」先發生，接著才引發「上層服務啟動失敗（果）」。因此，真正導致崩潰的真兇往往藏在最上面的錯誤訊息中，而不是最後一行的失敗宣告。**

```text
[因] nginx: [emerg] directive is not terminated by ";" in /etc/nginx/... :N
      ↓ 因為設定檔語法錯，所以啟動前的檢查沒過
[果] nginx.service: Control process exited, code=exited, status=1
      ↓ 因為檢查沒過，所以服務啟動失敗
[果] Failed to start nginx.service
```
新手常常盯著最後一行 `Failed to start` 發呆——那行只告訴你「它失敗了」，不告訴你「為什麼」。

**常見死因與分診路線：**

| log 裡出現 | 屬於哪一類 | 下一步 |
| :--- | :--- | :--- |
| `syntax error`、`is not terminated by ";"`、`Unable to locate executable` | 設定錯 | 路徑一 |
| `Permission denied`、`cannot open` | 權限錯 | 路徑二 |
| `No space left on device`、`Cannot allocate memory` | 資源不足 | 路徑三 |
| `Address already in use` | Port 被佔 | 下一階段 |

---

## 常見誤會
* **「log 太多了，根本讀不完。」**
    不用讀完。用 `-p err` 只撈錯誤，通常只剩三五行。讀 log 的能力不是讀得快，是知道怎麼過濾。
* **「紅色的、有顏色的才是錯誤。」**
    顏色只是終端機幫你標記的輔助。當你把 log 導到檔案、或用管線處理時，顏色會完全消失。你要練的是讀懂文字，不是找顏色。
* **「最後一行寫 Failed，那就是原因。」**
    最後一行通常是結果，不是原因。原因在更上面。往上找，找到第一個 error。

## 觀念確認（小任務）
log 裡有這兩行：

```text
Unable to locate executable '/usr/sbin/nginxXXX': No such file or directory
Main process exited, code=exited, status=203/EXEC
```

哪一行是根本原因？
* A. 第一行——找不到執行檔
* B. 第二行——程序結束，代碼 203
* C. 兩行都是原因

**建議答案是 A。** 因為找不到那個執行檔，所以程序才無法啟動、才回報 203。第二行是第一行造成的結果。修的時候要修第一行，修不了第二行。

> **✅ 分診進度：步驟 2 完成。**
> 遺言拿到了，它說設定檔第 N 行少了一個分號。
> 這屬於路徑一：設定錯。下一關，把它修好。
